仮想マシンの内部DNS解決

概要

本チュートリアルでは、OpenShift Virtualizationにおいて、クラスター内通信のために仮想マシンを安定したDNS名で探索可能にする方法を説明します。Kubernetes Serviceを使用してVMのDNSベースのサービス探索を有効にし、IPアドレスをハードコードすることなくPodからVMへ、またVMからVMへの通信を実現する方法を学びます。

学習目標

  • Kubernetes Serviceを使用してVMを公開する方法

  • VMのDNS解決をテストする方法

  • OpenShift VirtualizationでVMのDNSクエリがどのように解決されるか

前提条件

  • OpenShift VirtualizationオペレーターがインストールされたOpenShift 4.18以降

  • インストール済みのCLIツール: oc、virtctl

  • Kubernetes ServiceとDNSに関する基本的な知識

パート1: デプロイ、公開、テスト

ステップ 1: 名前空間の作成

このチュートリアル用の専用名前空間を作成します。

oc new-project myapp

ステップ 2: Fedora VMのデプロイ

Fedora VMをデプロイします。

この例のcloud-initパスワードはデモ目的のみのものです。本番環境では、常に強力で一意のパスワードまたはSSHキー認証を使用してください。

fedora-app-vm.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-app-vm
  namespace: myapp
  labels:
    app: myapp
spec:
  runStrategy: Always
  dataVolumeTemplates:
    - metadata:
        name: fedora-app-vm-volume
      spec:
        sourceRef:
          kind: DataSource
          name: fedora
          namespace: openshift-virtualization-os-images
        storage:
          resources:
            requests:
              storage: 30Gi
  template:
    metadata:
      labels:
        kubevirt.io/domain: fedora-app-vm
        app: myapp
    spec:
      domain:
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - name: default
              masquerade: {}
        resources:
          requests:
            memory: 2Gi
      networks:
        - name: default
          pod: {}
      volumes:
        - dataVolume:
            name: fedora-app-vm-volume
          name: rootdisk
        - cloudInitNoCloud:
            userData: |-
              #cloud-config
              user: fedora
              password: fedora
              chpasswd: { expire: false }
              ssh_pwauth: true
          name: cloudinitdisk
EOF

VMが実行状態になるまで待ちます。

watch 'oc get vm,vmi,dv -n myapp'

VMが Running かつ Ready: True になるまで待ち、完了したら Ctrl+C を押します。

ステップ 3: VM用Serviceの作成

VMをターゲットにするヘッドレスServiceを作成します。

headless-service.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
  namespace: myapp
spec:
  clusterIP: None
  selector:
    kubevirt.io/domain: fedora-app-vm
  ports:
    - name: ssh
      protocol: TCP
      port: 22
      targetPort: 22
EOF

ServiceとEndpointSliceを確認します。

oc get svc myapp-service -n myapp
oc get endpointslice -n myapp -l kubernetes.io/service-name=myapp-service

期待される出力:

Serviceは TYPE: ClusterIP で CLUSTER-IP: None (ヘッドレスサービス) と表示されるはずです。

EndpointSliceのENDPOINTS列には、VMのPod IPアドレスが表示されるはずです。

ステップ 4: DNS解決と接続性のテスト

DNS解決と接続性をテストするためにデバッグ用Podを作成します。

oc apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: debug-dns
  namespace: myapp
spec:
  containers:
  - name: debug
    image: registry.access.redhat.com/ubi9/ubi
    command: ["sleep", "3600"]
    securityContext:
      allowPrivilegeEscalation: false
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop:
        - ALL
      seccompProfile:
        type: RuntimeDefault
  restartPolicy: Never
EOF

# Podの準備ができるまで待機
oc wait --for=condition=Ready pod/debug-dns -n myapp --timeout=60s

# DNS解決をテスト
oc exec -n myapp debug-dns -- getent hosts myapp-service.myapp.svc.cluster.local

# クリーンアップ
oc delete pod debug-dns -n myapp

期待される結果:

  • DNS解決によってVMのPod IPアドレスが返されます（例: 10.128.0.169 myapp-service.myapp.svc.cluster.local）

これは、ServiceのDNS名がVMのIPアドレスに正しく解決されていることを確認するものです。

パート2: 仕組みの理解

DNS解決フロー

PodまたはVMがServiceをDNS名でクエリする場合、以下のフローが発生します。

┌───────────────────────────────────────────────────────────────┐
│ OpenShift Cluster                                             │
│                                                               │
│  [Client VM/Pod]                                              │
│         │                                                     │
│         │ 1. DNS Query: myapp-service.myapp.svc...            │
│         │                                                     │
│         ↓                                                     │
│  [/etc/resolv.conf]                                           │
│   nameserver 172.30.0.10                                      │
│   search myapp.svc.cluster.local svc.cluster.local            │
│         │                                                     │
│         │ 2. Send query to DNS server                         │
│         ↓                                                     │
│  ┌──────────────┐                                             │
│  │   CoreDNS    │                                             │
│  │ 172.30.0.10  │                                             │
│  └──────┬───────┘                                             │
│         │                                                     │
│         │ 3. Lookup Service record                            │
│         ↓                                                     │
│  ┌──────────────────────────────────────────────┐             │
│  │ Service: myapp-service                       │             │
│  │ Type: Headless                               │             │
│  │ Selector: kubevirt.io/domain=fedora-app-vm   │             │
│  └──────┬───────────────────────────────────────┘             │
│         │                                                     │
│         │ 4. Return VM Pod IP                                 │
│         ↓                                                     │

│         │                                                     │
│         │ 5. Connect to VM                                    │
│         ↓                                                     │

│  │  fedora-app-vm       │                                     │

│                                                               │
└───────────────────────────────────────────────────────────────┘

ステップバイステップの解説

ステップ 1: DNSクエリの開始

クライアント（他のVMやPod）が myapp-service.myapp.svc.cluster.local に接続する必要があります。クライアントがDNSクエリを開始します。

ステップ 2: CoreDNSへのクエリ送信

クライアントは /etc/resolv.conf ファイルを読み取り、そこに記載されているクラスターDNSサービス（通常は 172.30.0.10）にDNSクエリを送信します。

ステップ 3: CoreDNSによるServiceレコードの参照

CoreDNSはクラスター内のすべてのKubernetes Serviceのデータベースを保持しています。CoreDNSは myapp 名前空間の myapp-service というServiceを検索します。

ステップ 4: ServiceがVMのPod IPを返却

Serviceがヘッドレス（clusterIP: None）であるため、CoreDNSはEndpointSliceからVMのPodの実際のIPアドレスを返します。通常のClusterIP Serviceの場合は、代わりにServiceの仮想IPが返されます。

ステップ 5: 接続の確立

クライアントはIPアドレスを受け取り、そのVMへの直接接続を確立します。

主要なポイント

  • DNSにはServiceが必要: 仮想マシンはCoreDNSに自動的には登録されません。Kubernetes ServiceのみがDNSレコードを作成します。

  • VM名はDNS名ではない: VMの metadata.name はKubernetesのリソース識別子であり、DNS名ではありません。ネットワークアクセスには使用できません。

  • Service名がDNS名になる: Serviceの metadata.name が、<service-name>.<namespace>.svc.cluster.local としてCoreDNSに登録されます。

  • ヘッドレスサービスはVMのIPを返す: ヘッドレスService（clusterIP: None）はVMのPod IPを直接返すため、VM間での直接通信が可能になります。

  • 通常のサービスは負荷分散を提供する: 通常のClusterIP Serviceは安定した仮想IPを提供し、複数のVM間でトラフィックを分散できます。

  • 検索ドメインにより短い名前が使用可能: /etc/resolv.conf の search ドメインにより、同じ名前空間内のServiceにアクセスする際、完全修飾ドメイン名（FQDN）の代わりに myapp-service のような短い名前を使用できます。

一般的な問題

問題: DNS解決で結果が返されない

診断:

oc get svc myapp-service -n myapp
oc get endpointslice -n myapp -l kubernetes.io/service-name=myapp-service

一般的な原因:

  • ServiceのセレクターがVMのラベルと一致していない

  • VMが実行されていない（EndpointSliceにエントリがない）

  • 名前空間が間違っている

解決策:

  1. ServiceのセレクターがVMの kubevirt.io/domain ラベルと完全に一致しているか確認する

  2. VMが実行中であることを確認する: oc get vmi -n myapp

  3. ServiceとVMの両方が同じ名前空間にあることを確認する

問題: 名前空間をまたいだDNS解決が失敗する

診断:

# 別の名前空間にデバッグ用Podを作成
oc new-project test-namespace 2>/dev/null || true

oc apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: debug-cross-ns
  namespace: test-namespace
spec:
  containers:
  - name: debug
    image: registry.access.redhat.com/ubi9/ubi
    command: ["sleep", "300"]
    securityContext:
      allowPrivilegeEscalation: false
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop:
        - ALL
      seccompProfile:
        type: RuntimeDefault
  restartPolicy: Never
EOF

# 待機してテスト
oc wait --for=condition=Ready pod/debug-cross-ns -n test-namespace --timeout=60s
oc exec -n test-namespace debug-cross-ns -- \
  getent hosts myapp-service.myapp.svc.cluster.local

# クリーンアップ
oc delete pod debug-cross-ns -n test-namespace

一般的な原因:

  • FQDNではなく短い名前を使用している

  • ネットワークポリシーによって名前空間をまたぐトラフィックがブロックされている

解決策:

  1. 名前空間をまたぐ場合は常にFQDNを使用する: <service>.<namespace>.svc.cluster.local

  2. 制限的なNetworkPolicyが設定されていないか確認する
