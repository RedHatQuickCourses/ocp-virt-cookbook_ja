Layer 2 セカンダリネットワークの設定

概要

本ガイドでは、クラスター全体の論理スイッチを介して VirtualMachines を接続するための layer2 セカンダリネットワークの設定方法を説明します。

これを行うことで、VirtualMachines はデフォルトの Pod ネットワークをプライマリネットワークとして継続的に使用できます。これは、以下の条件が当てはまる場合に有効です。

  • 同じ名前空間内の（ノードが異なる場合でも）VM 間に単純な接続性が必要な場合。

  • 名前空間内の VM 間でネットワークの分離が不要な場合。

  • VM 間のネットワーク接続を複数の名前空間にまたがって拡張する必要がない場合。

  • クラスター内のノードの物理ネットワークインターフェイスへのアクセスが不要な場合。

  • インターネットからセカンダリネットワークへのイングレス接続をルーティングする必要がない場合。

  • 最後に、セカンダリネットワークから外部ネットワークへのアクセスが不要な場合。

検証済みバージョン:

OpenShift 4.20,4.21

前提条件

  • OpenShift Virtualization オペレーターがインストールされた OpenShift 4.20 以降のクラスター

  • NetworkAttachmentDefinition リソースを作成するためのクラスター管理者権限

  • CLI ツールのインストール: oc, virtctl

  • Kubernetes ネットワークの概念に関する基本的な理解

手順

プロセスは以下の3つのステップで構成されます。

  • 名前空間の作成

  • NetworkAttachmentDefinition の作成

  • VirtualMachines のネットワークへの接続

    • 最初の VirtualMachine の作成

    • 2つ目の VirtualMachine の作成

  • セカンダリネットワークのテスト

名前空間の作成

最初のステップとして、まだ作成していない場合は、ネットワークを定義する NetworkAttachmentDefinition と、ネットワークに接続したい VirtualMachines を格納するための名前空間を作成します。この場合、特別なラベルは必要ありません。

YAML を定義することなく、oc バイナリを使用して簡単に新しい名前空間を作成できます。

oc create namespace example-l2-secondary

新しい名前空間が作成されたので、現在の kube コンテキスト（現在作業中のクラスターと名前空間）を新しい名前空間（OpenShift 用語ではプロジェクト）に切り替えるのが最適です。

oc project example-l2-secondary

名前空間が作成され、現在のコンテキストが設定されたら、NetworkAttachmentDefinition の作成 に進みます。

NetworkAttachmentDefinition の作成

2番目のステップは、名前空間内に layer2 ネットワークを定義する NetworkAttachmentDefinition を作成することです。

l2-network.nad.yaml という名前の新しいファイルを作成し、以下の YAML 定義をファイルの内容としてコピー＆ペーストしてから、エディターを保存して閉じます。

apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: l2-network
  namespace: example-l2-secondary
spec:
  config: |-
    {
            "cniVersion": "0.3.1",
            "name": "l2-network",
            "type": "ovn-k8s-cni-overlay",
            "topology":"layer2",
            "mtu": 1400,
            "netAttachDefName": "example-l2-secondary/l2-network"
    }

oc apply を使用して、OpenShift クラスター上の定義済みの名前空間 example-l2-secondary に NetworkAttachmentDefinition を適用します。

oc apply -f l2-network.nad.yaml

NetworkAttachmentDefinition が正常に作成されたことを確認します。次に進む前に、l2-network が example-l2-secondary 名前空間に存在している必要があります。

oc get net-attach-def l2-network -n example-l2-secondary

l2-network が作成されたら、VirtualMachines の接続 に進みます。

VirtualMachines の接続

次のステップは、VirtualMachines を作成し、作成したばかりの layer2 ネットワークに接続することです。 これは、一般的な OpenShift (Kubernetes) の流儀に従い、VirtualMachine API オブジェクトを作成してクラスターに適用することで行います。

このセクションでは、どちらもセカンダリの l2-network に接続される2つの新しい VM を作成します。

最初の VirtualMachine の作成

以下は、最初の VirtualMachine の YAML 定義です（主要なフィールドについては後述します）。 内容を fedora-l2-secondary.vm.yaml という新しいファイルにコピー＆ペーストしてください。

警告: この例の cloud-init パスワードはデモ目的のみのものです。本番環境では必ず強力で一意のパスワードまたは SSH 鍵認証を使用してください。

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-l2-secondary
  labels:
    app: fedora-l2-secondary
spec:
  dataVolumeTemplates:
  - metadata:
      name: fedora-l2-secondary-root
    spec:
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
      storage:
        resources:
          requests:
            storage: 30Gi
  instancetype:
    name: u1.large
  preference:
    name: fedora
  runStrategy: Always
  template:
    metadata:
      labels:
        app: fedora-l2-secondary
    spec:
      domain:
        devices:
          interfaces:
          - name: default
            masquerade: {}
          - name: secondary
            bridge: {}
        resources: {}
      terminationGracePeriodSeconds: 180
      networks:
      - name: default
        pod: {}
      - name: secondary
        multus:
          networkName: l2-network
      volumes:
      - name: fedora-l2-secondary-root
        dataVolume:
          name: fedora-l2-secondary-root
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            user: fedora
            password: fedora
            chpasswd:
              expire: False
          networkData: |
            version: 2
            ethernets:
              enp2s0:
                addresses:
                - 192.168.20.10/24
上記の VM の例について

上記の例におけるフィールド（spec.template.spec に関連）について：

domain.devices.interfaces - インターフェイス定義のリストを受け入れ、それぞれに2つのフィールドが定義されています。

  • name - インターフェイス名を指定します。ここでは networks で関連付けられたネットワーク名に合わせて secondary と命名しました。

  • bridge または masquerade - セカンダリインターフェイスは bridge を使用してハイパーバイザー Pod のインターフェイスに接続されます。どちらのフィールドも空のオブジェクト ({}) を受け入れます。

networks - インターフェイスの接続に使用するネットワークをリストアップする場所です。interfaces と同様に、各ネットワークには2つのフィールドがあります。

  • name - default ネットワークと区別するために secondary というネットワーク名を適切に選択しました。

  • pod または multus - デフォルト以外のネットワークを選択するには、フィールド名として multus を指定します。デフォルトの pod ネットワークは空のオブジェクトを受け入れますが、multus は networkName サブフィールドを必要とします。

    • networkName (multus に必須) - 前述の手順で作成した NetworkAttachmentDefinition の名前である l2-network を指定します。

volumes - fedora-l2-secondary-root ディスクと、ユーザー資格情報やネットワークアドレスなどの VM 設定データを渡すために使用される cloudinitdisk の2つのボリュームが定義されています。

oc create を使用して、OpenShift クラスター上の現在の名前空間 (example-l2-secondary) に新しい VirtualMachine を作成できます。

oc create -f fedora-l2-secondary.vm.yaml

次に進む前に、fedora-l2-secondary VM が example-l2-secondary 名前空間で実行中であることを確認してください。Provisioning 状態の間は、コマンドを数回実行する必要がある場合があります。

oc get vm fedora-l2-secondary -n example-l2-secondary
トラブルシューティング

VirtualMachine が Pending 状態で停止しているように見える場合、VM のスケジューリングを妨げている要因があります。考えられる要因は以下の通りです。

  • クラスター上の基盤となる DataVolume のプロビジョニングに問題がある（ディスク容量不足など）。

  • l2-network の NetworkAttachmentDefinition が存在しない。

  • OpenShift クラスターにリソース制限がある。メモリ不足や vCPU 不足により、VM がスケジューリングできていない可能性があります。

問題の原因を特定するには、下位レベルのリソース（vmi、datavolume、pvc など）に対して oc describe または oc get -o yaml を使用してください。 エラーメッセージは、該当するリソースの Events (oc describe) または Status (oc get -o yaml) に記載されているはずです。

2つ目の VirtualMachine の作成

テストを行う前に、セカンダリネットワークを介した接続を確立するために2つ目の VM が必要です。

OpenShift クラスター上の現在の名前空間に VirtualMachine を適用します。

oc apply -f fedora-l2-secondary2.vm.yaml

fedora-l2-secondary2 という名前の2つ目の VM が example-l2-secondary 名前空間で実行中であることを確認してください。テスト を行う前に、両方の VirtualMachines が実行されている必要があります。

oc get vm fedora-l2-secondary2 -n example-l2-secondary

セカンダリネットワークのテスト

セカンダリネットワークは layer2 専用であり、クラスターと名前空間内の仮想ネットワークとしてのみ存在するため、ある VM から別の VM への接続性をテストする必要があります。

テストを進めるには、example-l2-secondary 名前空間に以下のリソースが存在している必要があります。

  • NetworkAttachmentDefinition: l2-network

  • VirtualMachine: fedora-l2-secondary (セカンダリ NIC IP アドレス: 192.168.20.10)

  • VirtualMachine: fedora-l2-secondary2 (セカンダリ NIC IP アドレス: 192.168.20.20)

最も簡単なテスト方法は、一方の VM（または両方の VM）のコンソールに接続し、もう一方の VM に ping を送信することです。

virtctl コマンドを使用して最初の VirtualMachine のコンソールにアクセスし、ユーザー名:`fedora`、パスワード:`fedora` でログインします（ログインプロンプトが表示されない場合は ENTER を押してください）。

virtctl console fedora-l2-secondary

VirtualMachine の IP アドレスを確認します。次のコマンドを実行すると、enp2s0 インターフェイスに対して静的アドレス 192.168.20.10 が返されるはずです。

ip addr show

最初の VM のコンソールから fedora-l2-secondary2 (IP アドレス: 192.168.20.20) VM に ping を送信します。応答があれば、セカンダリネットワークは機能しています（CTRL+C を押すと ping がキャンセルされます）。

ping 192.168.20.20

VM コンソールを終了するには、CTRL+] を押します。

この演習で作成したすべてリソースを削除するには、別の名前空間に切り替えて example-l2-secondary 名前空間を削除するだけです。

oc project default
oc delete namespace example-l2-secondary

まとめ

このチュートリアルでは、以下の内容を学びました。

  • ovn-k8s-cni-overlay CNI プラグインを使用して layer2 トポロジーの NetworkAttachmentDefinition を作成する方法

  • プライマリ masquerade インターフェイスとセカンダリ bridge インターフェイスの両方を持つ VirtualMachines を設定する方法

  • cloud-init networkData を使用してセカンダリネットワークに静的 IP アドレスを割り当てる方法

  • 同じ OVN 管理の論理スイッチ上の VM 間の Layer 2 接続を確認する方法

  • Linux ブリッジや localnet などの他のネットワークオプションと比較して、いつ Layer 2 セカンダリネットワークを使用するか
