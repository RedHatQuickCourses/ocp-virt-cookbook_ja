Cloud-init IP 設定トラブルシューティングガイド

本ガイドでは、OpenShift Virtualization における cloud-init を使用した IP アドレス設定に関する問題のトラブルシューティング手順、検証コマンド、および解決策を提供します。

目次

  • クイックヘルスチェック

  • cloud-init のデバッグ

  • シナリオ別の一般的な問題

  • 診断コマンド

クイックヘルスチェック

VM ステータスの確認

oc get vm <vm-name> -n <namespace>
oc get vmi <vm-name> -n <namespace>

VM が IP アドレスを持っているか確認

oc get vmi <vm-name> -n <namespace> -o jsonpath='{.status.interfaces[*].ipAddress}'

Guest Agent が実行中か確認

oc get vmi <vm-name> -n <namespace> -o jsonpath='{.status.guestOSInfo}'

空の場合、Guest Agent は実行されていません（Guest Agent の問題 を参照）。

cloud-init のデバッグ

VM コンソールへのアクセス

virtctl console <vm-name> -n <namespace>

cloud-init ステータスの確認

VM 内で実行:

# cloud-init の完了ステータスを確認
cloud-init status

# 成功時の期待される出力:
# status: done

# 詳細なステータスを確認
cloud-init status --long

cloud-init ログの確認

# メインの cloud-init ログ
cat /var/log/cloud-init.log

# 出力ログ (コマンドの実行結果を表示)
cat /var/log/cloud-init-output.log

# エラーを確認
grep -i error /var/log/cloud-init.log
grep -i failed /var/log/cloud-init.log

適用された cloud-init 設定の表示

# userdata を表示
sudo cat /var/lib/cloud/instance/user-data.txt

# ネットワーク設定を表示 (networkData を使用している場合)
sudo cat /var/lib/cloud/instance/network-config.txt

# cloud-init データを表示
sudo cat /var/lib/cloud/instance/cloud-config.txt

ネットワーク設定の確認

# ネットワークインターフェースを表示
ip addr show

# ルーティングテーブルを表示
ip route show

# DNS 設定を確認
cat /etc/resolv.conf

# NetworkManager を使用するシステム (Fedora, RHEL) の場合
nmcli device status
nmcli connection show

接続性のテスト

# ネットワーク接続のテスト
ping -c 3 8.8.8.8

# DNS 解決のテスト
nslookup google.com

# Pod ネットワークからテスト
curl http://<vm-ip>

シナリオ別の一般的な問題

シナリオ 1: DHCP 設定

問題: DHCP 経由で VM が IP を取得できない

症状:

$ oc get vmi fedora-dhcp-vm -n vms -o jsonpath='{.status.interfaces[0].ipAddress}'
# 空を返す、または IP が表示されない

診断:

# VM コンソールにアクセス
virtctl console fedora-dhcp-vm -n vms

# インターフェースステータスを確認
ip addr show enp1s0

# DHCP クライアントを確認
journalctl -u NetworkManager | grep -i dhcp

一般的な原因:

  • DHCP クライアントが実行されていない

  • ネットワークインターフェースがダウンしている

  • cloud-init の DHCP 設定エラー

解決策:

  1. VM 定義の cloud-init DHCP 設定を確認:

    cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      enp1s0:
        dhcp4: true

  2. ネットワークインターフェースを手動で再起動:

    sudo nmcli connection down "Wired connection 1"
sudo nmcli connection up "Wired connection 1"

  3. NetworkManager ログを確認:

    sudo journalctl -u NetworkManager --since "5 minutes ago"

問題: cloud-init 内のインターフェース名が間違っている

症状: ネットワーク設定が適用されず、ログに「device eth0 not found」と表示される。

診断:

# VM 内 - 実際のインターフェース名を確認
ip link show

一般的な原因:

  • ゲスト OS で使用されているインターフェース名ではなく、汎用名 (eth0, eth1) を使用している

解決策:

  1. ゲスト OS 内の実際のインターフェース名を確認してください。最新の Linux ディストリビューション (RHEL 8+, Fedora, Ubuntu 16.04+) では、予測可能な命名規則が使用されます。

    • 最初のインターフェース: enp1s0

    • 2番目のインターフェース: enp2s0

    • 3番目のインターフェース: enp3s0

  2. cloud-init の networkData を更新し、正しい名前を使用します:

    networkData: |
  version: 2
  ethernets:
    enp1s0:  # eth0 ではなく
      dhcp4: true

シナリオ 2: 静的 IP 設定

問題: 静的 IP が適用されない

症状: 設定した静的 IP ではなく、VM が DHCP IP を取得してしまう。

診断:

# VM 内
ip addr show enp1s0

# エラーがないか cloud-init ログを確認
grep -i "network" /var/log/cloud-init.log

一般的な原因:

  • networkData 内のインターフェース名が間違っている

  • DHCP が有効のままになっている

  • networkData YAML の構文エラー

  • cloud-init が networkData を処理していない

解決策:

  1. networkData の構文 (インデント、YAML 形式) を確認:

    networkData: |
  version: 2
  ethernets:
    enp1s0:
      dhcp4: false  # DHCP を無効化
      addresses:
        - 192.168.1.100/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4

  2. cloud-init のエラーを確認:

    cloud-init status --long
cat /var/log/cloud-init.log | grep -A 5 -B 5 "network"

  3. ネットワーク設定を手動で適用:

    sudo nmcli connection modify "Wired connection 1" \
  ipv4.method manual \
  ipv4.addresses 192.168.1.100/24 \
  ipv4.gateway 192.168.1.1 \
  ipv4.dns "8.8.8.8 8.8.4.4"

sudo nmcli connection up "Wired connection 1"

問題: 静的 IP は適用されたが接続できない

症状:

# VM 内
ip addr show enp1s0
# 正しい IP が表示される

ping 192.168.1.1
# Network unreachable

診断:

# ルーティングテーブルを確認
ip route show

# ゲートウェイに到達可能か確認
ping -c 1 192.168.1.1

一般的な原因:

  • ゲートウェイの設定漏れ、または誤り

  • サブネットマスクの誤り

  • 静的 IP 用に構成されていない物理ネットワーク

  • スイッチポートの VLAN 設定ミス

解決策:

  1. cloud-init 設定でゲートウェイを確認

  2. サブネットマスクがネットワーク設定と一致しているか確認

  3. 物理ネットワークがこのサブネット上で静的 IP を許可しているか確認

  4. VLAN を使用している場合は、NetworkAttachmentDefinition (NAD) とブリッジマッピングが正しいか確認

シナリオ 3: 複数のネットワークインターフェース

問題: セカンダリインターフェースが設定されない

症状: プライマリインターフェースのみ IP を取得し、セカンダリインターフェースがダウンしているか未設定の状態。

診断:

# VM 内
ip addr show

# NetworkManager 接続を確認
nmcli connection show

一般的な原因:

  • networkData にセカンダリインターフェースが定義されていない

  • セカンダリインターフェースの名前が間違っている

  • セカンダリネットワーク接続が機能していない

解決策:

  1. VM が複数のネットワーク接続を持っているか確認:

    spec:
  template:
    spec:
      networks:
        - name: default
          pod: {}
        - name: secondary-net
          multus:
            networkName: my-nad

  2. networkData 内のインターフェース両方を確認:

    networkData: |
  version: 2
  ethernets:
    enp1s0:
      dhcp4: true
    enp2s0:  # セカンダリインターフェース
      dhcp4: false
      addresses:
        - 192.168.100.10/24

  3. NAD が存在し、正しく設定されているか確認:

    oc get network-attachment-definition -n <namespace>
oc describe network-attachment-definition <nad-name> -n <namespace>

問題: デフォルトルートの競合

症状: 両方のインターフェースにデフォルトルートがあり、ルーティングの問題が発生している。

診断:

# VM 内
ip route show

# 複数のデフォルトルートが表示される
# default via 10.128.0.1 dev enp1s0
# default via 192.168.100.1 dev enp2s0

一般的な原因:

  • 両方のインターフェースがデフォルトゲートウェイで構成されている

  • DHCP がプライマリインターフェースにデフォルトルートを追加している

  • 静的設定がセカンダリインターフェースにデフォルトルートを追加している

解決策:

  1. networkData でセカンダリインターフェースからデフォルトルートを削除:

    enp2s0:
  dhcp4: false
  addresses:
    - 192.168.100.10/24
  # 'routes' セクションなし - デフォルトゲートウェイなし

  2. 両方にルートが必要な場合は、メトリックを使用して優先順位を付けます:

    enp1s0:
  dhcp4: true
  dhcp4-overrides:
    route-metric: 100  # 数値が低い方が優先される

enp2s0:
  dhcp4: false
  addresses:
    - 192.168.100.10/24
  routes:
    - to: default
      via: 192.168.100.1
      metric: 200  # 数値が高い = 優先順位が低い

シナリオ 4: VLAN インターフェースの設定

問題: VLAN インターフェースが作成されない

症状: VLAN インターフェース (例: enp2s0.100) が存在しない。

診断:

# VM 内
ip link show

# cloud-init ログを確認
grep -i vlan /var/log/cloud-init.log

一般的な原因:

  • networkData の VLAN 設定構文エラー

  • 親インターフェースがアップしていない

  • VLAN モジュールがロードされていない

解決策:

  1. networkData の VLAN 設定を確認:

    networkData: |
  version: 2
  ethernets:
    enp2s0: {}  # 親インターフェースを定義する必要がある
  vlans:
    vlan100:
      id: 100
      link: enp2s0
      addresses:
        - 192.168.100.10/24

  2. VLAN インターフェースを手動で作成:

    sudo nmcli connection add type vlan \
  con-name vlan100 \
  ifname enp2s0.100 \
  dev enp2s0 \
  id 100 \
  ip4 192.168.100.10/24

問題: VLAN インターフェースに接続性がない

症状: VLAN インターフェースは存在するがネットワークに到達できない。

診断:

# VLAN インターフェースのステータスを確認
ip addr show enp2s0.100

# トラフィックがタグ付けされているか確認
tcpdump -i enp2s0 -n vlan 100

一般的な原因:

  • 物理スイッチポートがトランクモードになっていない

  • スイッチトランクで VLAN が許可されていない

  • 設定の VLAN ID が間違っている

  • NAD が VLAN 用に設定されていない

解決策:

  1. スイッチポートがトランクモードで、VLAN 100 を許可しているか確認

  2. NAD の VLAN 設定を確認:

    oc get network-attachment-definition <nad-name> -n <namespace> -o yaml

    "vlan": 100 があるか確認してください。

  3. localnet 用の OVN ブリッジマッピングを確認:

    oc get nncp -A

シナリオ 5: IPv6 設定

問題: IPv6 アドレスが適用されない

症状: IPv4 のみ設定され、IPv6 アドレスがない。

診断:

# VM 内
ip -6 addr show

一般的な原因:

  • cloud-init で IPv6 が有効になっていない

  • OS で IPv6 が無効になっている

解決策:

  1. networkData で IPv6 を有効化:

    networkData: |
  version: 2
  ethernets:
    enp1s0:
      dhcp4: true
      dhcp6: true
      # または静的 IPv6:
      addresses:
        - 2001:db8::10/64

  2. IPv6 が無効になっていないことを確認:

    cat /proc/sys/net/ipv6/conf/all/disable_ipv6
# 0 (有効) である必要があります

シナリオ 6: DNS 設定

問題: DNS が機能しない

症状:

# VM 内
ping 8.8.8.8
# 成功

ping google.com
# Unknown host

診断:

cat /etc/resolv.conf

# DNS 解決を確認
nslookup google.com

一般的な原因:

  • cloud-init でネームサーバーが設定されていない

  • DHCP が DNS サーバーを提供していない

  • /etc/resolv.conf が空、または間違っている

解決策:

  1. networkData にネームサーバーを追加:

    networkData: |
  version: 2
  ethernets:
    enp1s0:
      dhcp4: false
      addresses:
        - 192.168.1.100/24
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
        search:
          - example.com

  2. DNS を手動で設定:

    echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf

インターフェース命名リファレンス

VM 内の最新 Linux ディストリビューションは、PCI スロット位置に基づいた予測可能なインターフェース名を使用します。

ネットワーク接続順序 インターフェース名 一般的な用途

1番目 (デフォルト Pod ネットワーク)

enp1s0

プライマリネットワーク (DHCP)

2番目 (Multus/NAD)

enp2s0

セカンダリネットワーク

3番目 (Multus/NAD)

enp3s0

3番目のネットワーク

4番目 (Multus/NAD)

enp4s0

4番目のネットワーク

ゲスト OS に表示されるインターフェース名を使用してください。最新の Linux ディストリビューション (RHEL 8+、Fedora、Ubuntu 16.04+) は、上記に示す enpXsY 形式を使用します。古いディストリビューションでは、eth0、eth1 などが使用される場合があります。

cloud-init バージョンの違い

cloud-init v1 vs v2 ネットワーク設定

Version 1 (旧式、レガシー):

networkData: |
  version: 1
  config:
    - type: physical
      name: eth0
      subnets:
        - type: dhcp

Version 2 (最新、推奨):

networkData: |
  version: 2
  ethernets:
    enp1s0:
      dhcp4: true

推奨: 新しい設定には必ずバージョン 2 を使用してください。

[fedora@fedora ~]$ cloud-init status
status: done
