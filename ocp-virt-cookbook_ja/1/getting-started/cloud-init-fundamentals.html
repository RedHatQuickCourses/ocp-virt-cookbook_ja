OpenShift Virtualization VMのためのCloud-init基礎

概要

Cloud-initは、仮想マシンの初回起動時の初期化を自動化するための業界標準ツールです。 OpenShift Virtualizationでは、ほぼすべてのVMがCloud-initを使用して、ユーザー設定、パスワード設定、SSHキーの注入、パッケージのインストール、およびネットワークアドレスの割り当てを行います。 本チュートリアルでは、Cloud-initの動作原理、KubeVirtによるVMへのデータ配信方法、そしてVirtualMachineマニフェストへのCloud-init設定の埋め込み方法について解説します。

  • Cloud-initが正常に実行されたことを確認する方法

Cloud-initとは？

Cloud-initは、仮想マシンの初回起動時にゲストOS内で実行されるオープンソースフレームワークです。 外部ソースから設定データを読み込み、ユーザーアカウントの作成、SSHアクセスの設定、パッケージのインストール、設定ファイルの書き込み、スクリプトの実行など、本来手動で行う必要のあるタスクを自動化するために使用されます。

Cloud-initは、Fedora、RHEL、CentOS、Ubuntu、Debianなど、ほとんどのLinuxディストリビューションでサポートされています。 これは、AWS、Azure、GCPなどの主要なクラウドプラットフォームや、OpenShift Virtualizationの基盤であるKubeVirtを含む仮想化プラットフォーム全体で標準的な初期化メカニズムとなっています。

注意: Cloud-initはデフォルトではVMの初回起動時にのみ実行されます。インスタンスデータが明示的にクリアされない限り、2回目以降の再起動ではCloud-initはスキップされます。

KubeVirtによるCloud-initの配信

従来のクラウド環境では、VMはメタデータサービス（169.254.169.254 のHTTPエンドポイント）からCloud-initデータを取得します。 KubeVirtは、*NoCloudデータソース*と呼ばれる異なるアプローチを採用しています。

NoCloudを使用する場合、KubeVirtは以下の動作を行います。

  1. VirtualMachineマニフェスト（または参照されたSecret）からCloud-initデータを読み取る

  2. データをISOイメージとしてフォーマットされた仮想ディスクにパッケージ化する

  3. virtioデバイスを介してディスクをVMに接続する

  4. ゲストOS内のCloud-initがそのディスクを検出し、データを読み取って設定を適用する

つまり、Cloud-initの設定はKubernetesリソース内に完結しており、外部のメタデータサービスは不要です。

spec:
  template:
    spec:
      volumes:
        - name: cloudinitdisk
          cloudInitNoCloud:        (1)
            userData: |
              #cloud-config
              ssh_pwauth: true
              user:
                name: fedora
                lock_passwd: false
                plain_text_passwd: changeme
1 cloudInitNoCloud ボリュームタイプを指定することで、KubeVirtにNoCloudデータソースディスクを作成するよう指示します。

Cloud-initのデータタイプ

Cloud-initは3種類の構成データを受け入れます。KubeVirtでは、これらは cloudInitNoCloud ボリューム下のフィールドに対応します。

userData (cloud-config)

主要な構成チャネルです。userDataには cloud-config ドキュメントが含まれます。これは #cloud-config ヘッダーで始まるYAMLファイルです。ここでユーザー、パッケージ、スクリプト、ファイル、およびVMのほとんどのカスタマイズを定義します。

cloudInitNoCloud:
  userData: |
    #cloud-config
    ssh_pwauth: true
    user:
      name: fedora
      lock_passwd: false
      plain_text_passwd: s3cur3p4ss
      ssh_authorized_keys:
        - ssh-ed25519 AAAA... user@workstation
    packages:
      - vim
      - tmux

重要: 1行目の #cloud-config ヘッダーは必須です。これがないと、Cloud-initはデータをcloud-configドキュメントとして認識せず、処理を黙ってスキップします。

networkData (network-config v2)

VMインターフェース用のネットワーク設定です。Netplan互換の network-config version 2 形式を使用します。静的IP、カスタムDNS、またはマルチインターフェース設定が必要な場合に使用します。

シンプルなDHCP設定の例：

cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      enp1s0:
        dhcp4: true

DNSとルートを指定した静的IP設定の例：

cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      enp1s0:
        addresses:
          - 192.168.1.100/24             (1)
        nameservers:
          addresses:
            - 192.168.1.1
            - 8.8.8.8                    (2)
        routes:
          - to: default
            via: 192.168.1.1             (3)
          - to: 10.0.0.0/8
            via: 192.168.1.254           (4)
1 CIDR表記でのサブネットマスク付きの静的IPアドレス。
2 DNSリゾルバのアドレス。
3 ルートとして定義されたデフォルトゲートウェイ（gateway4 は非推奨）。
4 特定のゲートウェイ経由で 10.0.0.0/8 ネットワークに到達するための追加ルート。

インターフェース名が予測できない場合に備え、名前の代わりにMACアドレスでインターフェースを指定することも可能です。

cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      primary:
        match:
          macaddress: "52:54:00:12:34:56"
        addresses:
          - 192.168.1.100/24
        routes:
          - to: default
            via: 192.168.1.1

注意: KubeVirt VMでは、最初のインターフェースは通常 enp1s0、2番目は enp2s0 となります。これらの名前はNetworkAttachmentDefinitionの順序ではなく、VirtualMachineスペックでインターフェースが定義された順序に対応します。

マルチインターフェース設定やVLAN、高度なシナリオについては、静的IP設定を参照してください。

metadata

instance-id や local-hostname などのVMインスタンスメタデータです。多くの場合、KubeVirtはVirtualMachineリソースからメタデータを自動的に生成するため、手動で設定する必要はほとんどありません。

cloudInitNoCloud:
  metadata: |
    instance-id: my-vm-001
    local-hostname: my-vm

ヒント: ホスト名のみを設定したい場合は、カスタムメタデータを提供するのではなく、userData内の hostname モジュールを使用してください。

Cloud-initの実行ステージ

Cloud-initは起動プロセス中に5つのステージで実行されます。これらのステージを理解すると、設定がいつ反映されるかを予測するのに役立ちます。

ステージ 実行タイミング 動作

Generator

非常に早い起動時 (systemd generator)

Cloud-initを実行するかどうかを判断

Local

ネットワークが立ち上がる前

networkDataからネットワーク設定を適用

Network

ネットワークが利用可能になった後

残りのデータソース情報を取得し、ホスト名を設定

Config

Networkステージの後

設定モジュールの実行：ユーザー作成、ファイル書き込み、SSHキーのインストール

Final

起動の終了付近

runcmd コマンドの実行、パッケージのインストール、最終スクリプトの実行

この順序により、ネットワーク設定（networkData）はユーザー設定（userDataモジュール）よりも先に適用されます。runcmd で指定されたコマンドは、他のすべてのモジュールが完了した後に最後に実行されます。

一般的なCloud-configモジュール

以下のモジュールは、最も一般的なVM初期化タスクをカバーしています。各例は #cloud-config userDataドキュメントの該当セクションを示しています。

ユーザー作成とパスワード設定

#cloud-config
ssh_pwauth: true
user:
  name: fedora
  lock_passwd: false
  plain_text_passwd: s3cur3p4ss
  ssh_authorized_keys:
    - ssh-ed25519 AAAA...key-data... user@workstation

user キーはデフォルトユーザーを設定します。パスワードログインを許可するには lock_passwd: false と plain_text_passwd を設定します。最上位レベルで ssh_pwauth: true を追加すると、SSHでのパスワード認証が有効になります（多くのクラウドイメージではデフォルトで無効になっています）。

デフォルト以外の追加ユーザーを作成するには、users リストを使用します。

#cloud-config
users:
  - name: admin
    groups: wheel
    sudo: ALL=(ALL) NOPASSWD:ALL
    ssh_authorized_keys:
      - ssh-ed25519 AAAA...key-data...
  - name: developer
    shell: /bin/bash

パッケージのインストール

#cloud-config
package_update: true
packages:
  - vim
  - git
  - tmux
  - python3

注意: パッケージのインストールは final ステージで実行されます。VMはパッケージのインストール完了前であっても、コンソールからアクセス可能です。

コマンドの実行

#cloud-config
runcmd:
  - systemctl enable --now cockpit.socket
  - echo "VM initialized at $(date)" >> /var/log/vm-init.log

コマンドは final ステージ中に順番に実行されます。各エントリは文字列（/bin/sh 経由で実行）またはリスト（直接実行）にできます。

ファイルの書き込み

#cloud-config
write_files:
  - path: /etc/motd
    content: |
      Welcome to the OpenShift Virtualization VM.
      Managed by cloud-init.
    permissions: '0644'
  - path: /etc/myapp/config.yaml
    content: |
      database: postgres
      port: 5432
    owner: root:root
    permissions: '0600'

ヒント: write_files によって書き込まれたファイルは config ステージ中に作成されるため、runcmd が実行される前に存在します。これにより、設定ファイルを安全に書き込んでから、それに依存するサービスを runcmd で開始できます。

VirtualMachineマニフェストへのCloud-init埋め込み

VirtualMachineにCloud-initデータを提供するには2つのパターンがあります。

インライン記述 (cloudInitNoCloud)

最も簡単な方法は、Cloud-initデータをVirtualMachineマニフェストに直接埋め込む方法です。

minimal-cloud-init-vm.yaml をダウンロード

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-cloud-init-demo
  namespace: demo
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        app: fedora-cloud-init-demo
    spec:
      domain:
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - masquerade: {}
              name: default
        resources:
          requests:
            memory: 2Gi
      networks:
        - name: default
          pod: {}
      volumes:
        - dataVolume:
            name: fedora-cloud-init-demo
          name: rootdisk
        - cloudInitNoCloud:                    (1)
            userData: |                        (2)
              #cloud-config
              ssh_pwauth: true
              user:
                name: fedora
                lock_passwd: false
                plain_text_passwd: s3cur3p4ss
                ssh_authorized_keys:
                  - 'ssh-ed25519 AAAA...your-key... user@workstation'
              users:
                - name: userone
                  lock_passwd: false
                  plain_text_passwd: s3cur3p4ss
                  ssh_authorized_keys:
                    - 'ssh-ed25519 AAAA...your-key... user@workstation'
              packages:
                - vim
                - git
              runcmd:
                - echo "Cloud-init complete" > /tmp/cloud-init-done
          name: cloudinitdisk
  dataVolumeTemplates:
    - metadata:
        name: fedora-cloud-init-demo
      spec:
        sourceRef:
          kind: DataSource
          name: fedora
          namespace: openshift-virtualization-os-images
        storage:
          resources:
            requests:
              storage: 30Gi
1 cloudInitNoCloud ボリュームタイプがNoCloudデータソースディスクを作成します。
2 userData フィールドに #cloud-config で始まるcloud-config YAMLを含めます。

このパターンは小規模な設定に適しています。大規模な設定や機密情報を含む場合は、Secret参照を使用してください。

Secret参照

複雑な設定や、パスワードやトークンなどの機密値がCloud-initに含まれる場合は、Kubernetes Secretにデータを格納し、VirtualMachineからそれを参照します。

cloud-init-secret-vm.yaml をダウンロード

まず、Secretを作成します。

oc create secret generic my-vm-cloud-init \
  --from-literal=userdata='#cloud-config
ssh_pwauth: true
user:
  name: fedora
  lock_passwd: false
  plain_text_passwd: s3cur3p4ss
  ssh_authorized_keys:
    - ssh-ed25519 AAAA...your-key... user@workstation
users:
  - name: userone
    lock_passwd: false
    plain_text_passwd: s3cur3p4ss
    ssh_authorized_keys:
      - ssh-ed25519 AAAA...your-key... user@workstation
packages:
  - vim
  - git
' \
  -n demo

次に、VirtualMachineマニフェストから参照します。

volumes:
  - cloudInitNoCloud:
      secretRef:
        name: my-vm-cloud-init         (1)
    name: cloudinitdisk
1 KubeVirtはSecretから userdata キーを読み取り、それをCloud-initのuserDataとして提供します。

Cloud-initの実行確認

VMが起動した後、Cloud-initの設定処理が成功したかを確認します。

VMコンソールにアクセスします。

virtctl console fedora-cloud-init-demo -n demo

Cloud-initの状態を確認します。

cloud-init status --long

成功した場合の期待される出力：

status: done
extended_status: done
boot_status_code: enabled-by-generator
detail:
DataSourceNoCloud [seed=/dev/vdb]

Cloud-initがエラーを報告している場合は、ログを確認します。

cat /var/log/cloud-init.log | tail -50

詳細な診断手順や一般的な問題の解決策については、Cloud-initのトラブルシューティングを参照してください。

よくある間違い

#cloud-config ヘッダーの欠落

userDataの1行目は、正確に #cloud-config である必要があります。これがないと、Cloud-initはエラーメッセージを出さずにuserDataブロック全体を無視します。

YAMLのインデントエラー

Cloud-configはYAMLです。インデントの不整合（タブとスペースの混在、ネストの深さ間違い）は解析失敗の原因となります。VM内で cloud-init devel schema --config-file /path/to/config.yaml を使用して構文を検証してください。

ネットワーク設定をuserDataに記述

ネットワークインターフェースの設定は userData ではなく networkData に記述します。userData内にnetwork-config v2のYAMLを配置しても反映されません。

再起動時にCloud-initが再実行されるという期待

Cloud-initは初回起動時に一度だけ実行されます。再実行させるには、cloud-init clean を実行してから再起動する必要があります。

Secretキー名の間違い

secretRef を使用する場合、Secretは userdata という名前のキー（すべて小文字）を持つ必要があります。userData や user-data という名前のキーは検出されません。

Windows VMの場合：Sysprep

Cloud-initはLinux専用のツールです。Windows VMの場合、OpenShift Virtualizationは Sysprep をサポートしています。これはMicrosoft System Preparationツールのことで、初回起動時の同様のカスタマイズ（ホスト名、管理者パスワード、ドメイン参加、無人セットアップ）を実行します。

Windows VMの設定については、Windows VMへのリモートアクセスを参照してください。

クリーンアップ

本チュートリアルで作成したデモ用リソースを削除します。

oc delete vm fedora-cloud-init-demo -n demo
oc delete vm fedora-cloud-init-secret-demo -n demo
oc delete secret my-vm-cloud-init -n demo
oc delete namespace demo

まとめ

本チュートリアルでは以下の内容を学びました。

  • Cloud-initは標準のLinux VM初期化フレームワークであり、初回起動時に自動的に設定を行う

  • KubeVirtはNoCloudデータソースを介してCloud-initデータを配信し、VMに接続されたvirtioディスクとしてパッケージ化する

  • 3つのデータタイプ：userData（ユーザー・パッケージ・スクリプト用のcloud-config）、networkData（network-config v2）、metadata（インスタンス識別子）

  • Cloud-initはステージ順に実行される：ネットワーク設定が最初、次にユーザー/ファイル設定、最後にコマンドとパッケージ

  • 2つのマニフェストパターン：シンプルな構成にはインライン cloudInitNoCloud、複雑なデータや機密データにはSecret参照

  • cloud-init status --long および /var/log/cloud-init.log を使用した検証
