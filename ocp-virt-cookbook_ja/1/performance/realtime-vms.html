リアルタイム仮想マシン

概要

本チュートリアルでは、OpenShift Virtualizationでリアルタイム仮想マシン（VM）を構成する方法を解説します。リアルタイムVMは、5G RAN (vDU/vCU)、産業用制御システム、金融の取引処理（tick-to-trade）、高頻度テレメトリなど、厳格で境界の定められたレイテンシー要件を持つワークロード向けに設計されています。

共有CPUホスト上の標準的なVMでは、数十ミリ秒のスケジューリングレイテンシーのスパイクが発生することがあります。適切に構成されたリアルタイムVMは、複数のチューニング層を組み合わせることで、最悪の場合のレイテンシーを数マイクロ秒の範囲まで低減します。

構成レイヤー 排除されるジッターのソース 設定場所

専用CPUの配置

他ワークロードによるCPUの窃取（CPU steal）

VMスペック (dedicatedCpuPlacement)

エミュレーターのスレッド分離

vCPUコア上のQEMUのハウスキーピング

VMスペック (isolateEmulatorThread)

NUMAパススルー

NUMA間メモリアクセスのレイテンシー

VMスペック (guestMappingPassthrough)

Hugepages（ヒュージページ）

TLBミスおよびページテーブルウォーク

VMスペック (memory.hugepages)

ホストのRTカーネル

カーネルプリエンプションの遅延、タイマージッター

PerformanceProfile (realTimeKernel)

ホストのCPU分離

IRQおよびカーネルスレッドの干渉

PerformanceProfile (cpu.isolated)

ゲストのRTスケジューリング

ゲストvCPUのスケジューリング優先度

VMスペック (cpu.realtime)

バルーン/グラフィックスの無効化

不要なデバイス割り込み

VMスペック (autoattachMemBalloon: false)

本チュートリアルは、パフォーマンスシリーズの3番目のチュートリアルです。専用CPUの配置およびNUMA/Hugepagesのチュートリアルに基づいています。

前提条件

  • OpenShift VirtualizationオペレーターがインストールされたOpenShift 4.18以上（4.21で検証済み）

  • クラスター管理者アクセス権（system:admin または同等の権限）

  • oc および virtctl CLIのインストール

  • 少なくとも8つの物理CPUコアと16GiのRAMを備えたワーカーノード

  • CPUマネージャーの静的ポリシーが有効であること（専用CPUの配置チュートリアルを参照）

  • ヒュージページとNUMAトポロジーに関する知識（NUMAとHugepagesチュートリアルを参照）

  • Node Tuningオペレーターがインストールされていること（OpenShiftにデフォルトで含まれています）

リアルタイム要件の理解

仮想化におけるリアルタイムとは、平均応答速度が最も速いことではなく、*最悪の場合のレイテンシーが一定の範囲に収まっていること*を意味します。リアルタイムVMは、負荷がかかっている状態であっても、特定の操作が決定論的な時間枠内で完了することを保証します。

これを実現するには、ホストカーネル、ハイパーバイザー、ゲストカーネル、アプリケーションというスタック全体を通じて、無制限のジッターの原因をすべて排除する必要があります。概要の構成表は、各VMスペックのフィールドと、それが対応する特定のジッターソースを示しています。

リアルタイムVMは、最も脆弱なレイヤーと同程度の性能しか発揮できません。単一のレイヤーでも省略すると（例：RTカーネルを使用するがCPU分離を行わない）、ジッターの原因が残存し、目的が果たせなくなります。

ステップ 1: リアルタイム用MachineConfigPoolの準備

リアルタイムノードには専用のMachineConfigPoolが必要です。これにより、他のノードに影響を与えることなく、PerformanceProfileによってRTカーネル、CPU分離、およびヒュージページ構成を適用できます。

まず、対象のワーカーノードに worker-rt ロールを付与します。

oc label node <node-name> node-role.kubernetes.io/worker-rt=

worker ロールと worker-rt ロールの両方を持つノードを選択するMachineConfigPoolを作成します。

oc apply -f - <<EOF
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfigPool
metadata:
  name: worker-rt
  labels:
    machineconfiguration.openshift.io/role: worker-rt
spec:
  machineConfigSelector:
    matchExpressions:
    - key: machineconfiguration.openshift.io/role
      operator: In
      values:
      - worker
      - worker-rt
  nodeSelector:
    matchLabels:
      node-role.kubernetes.io/worker-rt: ""
EOF

プールが作成され、ノードがリストに含まれていることを確認します。

oc get mcp worker-rt

想定される出力:

NAME        CONFIG   UPDATED   UPDATING   DEGRADED   MACHINECOUNT   READYMACHINECOUNT
worker-rt            True      False      False      1              1

ステップ 2: PerformanceProfileの適用

PerformanceProfileリソースは、Node Tuningオペレーターを構成し、以下の操作を行います。

  • リアルタイムカーネル (kernel-rt) のインストール

  • ワークロード用のCPU 2-7の分離（カーネルスレッドやIRQのスケジューリングを禁止）

  • システムサービス用のCPU 0-1の予約

  • 4 x 1Giヒュージページの事前割り当て

  • レイテンシーを最小化するカーネル起動パラメーターの適用

performance-profile.yamlをダウンロード

oc apply -f - <<EOF
apiVersion: performance.openshift.io/v2
kind: PerformanceProfile
metadata:
  name: realtime-workers
spec:
  cpu:
    isolated: "2-7"
    reserved: "0-1"
  hugepages:
    defaultHugepagesSize: 1G
    pages:
    - size: 1G
      count: 4
  realTimeKernel:
    enabled: true
  additionalKernelArgs:
  - nmi_watchdog=0
  - audit=0
  - mce=off
  - processor.max_cstate=1
  - idle=poll
  - intel_idle.max_cstate=0
  nodeSelector:
    node-role.kubernetes.io/worker-rt: ""
  machineConfigPoolSelector:
    machineconfiguration.openshift.io/role: worker-rt
EOF
シングルノードOpenShift (SNO) の場合は、ステップ1のMachineConfigPool作成をスキップしてください。代わりに、nodeSelector を node-role.kubernetes.io/master: "" に、machineConfigPoolSelector を pools.operator.machineconfiguration.openshift.io/master: "" に設定して、既存のマスタープールを直接ターゲットにします。machineConfigPoolSelector は、内部で使用される machineConfigSelector ではなく、MCP自体の metadata.labels にあるラベルと一致させる必要があります。VMマニフェストの nodeSelector でそのラベルを使用するため、ステップ1の worker-rt ラベル (oc label node <node-name> node-role.kubernetes.io/worker-rt=) を付与しておく必要があります。
PerformanceProfile APIでは、ヒュージページサイズにIEC単位 (1Gi, 2Mi) ではなくSI単位 (1G, 2M) を使用します。これは、VMスペックで使用されるKubernetesリソース構文（1Gi, 2Mi）とは異なります。

PerformanceProfileは、基礎となる複数のリソースを生成します。それらは Tuned プロファイル、カーネル引数用の MachineConfig、およびCPUマネージャー設定用の KubeletConfig です。その後、Machine Configオペレーターがターゲットノードを再起動します。

PerformanceProfileを適用すると、RTカーネルのインストールとカーネルパラメーターの適用のため、ターゲットノードが再起動されます。影響を受けるノードのダウンタイムを計画してください。

ロールアウトを監視します。

oc get mcp worker-rt -w

SNOの場合は、代わりに master プールを監視します。

oc get mcp master -w

UPDATED が True になり、READYMACHINECOUNT が MACHINECOUNT と一致するまで待ちます。ノードがRTカーネルをダウンロードし、再起動し、クラスターに再参加するため、10〜20分かかる場合があります。MCPは再起動完了後も UPDATING: True 状態のままになることがありますが、これはノードが構成の調整を完了し、クラスターに準備完了を報告している間です。ロールアウトが完全に完了するまで先に進まないでください。

カーネルパラメーターの理解

パラメーター 目的

nmi_watchdog=0

NMIウォッチドッグタイマーを無効化し、定期的な割り込みを削減

audit=0

カーネル監査を無効化し、監査に関連するレイテンシーを回避

mce=off

マシンチェック例外ログを無効化し、割り込み処理を削減

processor.max_cstate=1

CPUが深いスリープ状態（C-states > C1）になるのを防ぎ、復帰レイテンシーを回避

idle=poll

アイドルCPUに対して停止ではなくポーリングを使用し、消費電力と引き換えに低い復帰レイテンシーを実現

intel_idle.max_cstate=0

IntelのC-stateドライバーを無効化し、processor.max_cstate を補完
これらのパラメーターは、電力とレイテンシーの間でアグレッシブなトレードオフを行います。RTノード上のCPUは、アイドル時であっても最大電力を消費します。レイテンシーの影響を受けやすいワークロード専用のノードにのみ適用してください。

ステップ 3: RTノードが準備完了であることの確認

MachineConfigPoolのロールアウトが完了したら、ノード構成を確認します。

カーネルバージョンを確認します。

oc debug node/<node-name>

デバッグシェル内にて：

chroot /host uname -r

カーネル名に rt が含まれているはずです。例：

5.14.0-427.55.1.el9_4.x86_64+rt

カーネル起動パラメーターを確認して、CPU分離が構成されていることを確認します。

chroot /host cat /proc/cmdline | grep -o 'isolcpus=managed_irq,[^ ]*'

想定される出力:

isolcpus=managed_irq,2-7

分離されたCPUでティックレス (nohz) モードがアクティブであることを確認します。

chroot /host cat /sys/devices/system/cpu/nohz_full

想定される出力（PerformanceProfileの cpu.isolated と一致）:

2-7

デフォルトのIRQアフィニティマスクを確認します。

chroot /host cat /proc/irq/default_smp_affinity

managed_irq モードでは、デフォルトのアフィニティマスクがすべてのCPUを表示する可能性があります。IRQステアリングは静的なアフィニティマスクではなく、Tunedによって生成された irqbalance ヒントによって動的に処理されます。

ヒュージページの割り当てを確認します。

chroot /host grep -i hugepages /proc/meminfo

HugePages_Total が要求された数（1Giページの場合は4）と一致することを確認します。

kubevirt.io/realtime ラベルが存在することを確認します。

exit
oc get node <node-name> -L kubevirt.io/realtime

KubeVirtは、kernel.sched_rt_runtime_us が -1（無制限のRTスケジューリング）に設定されているノードに自動的にラベルを付けます。ラベルがない場合は、設定を確認してください。

oc debug node/<node-name> -- chroot /host sysctl kernel.sched_rt_runtime_us

想定される出力:

kernel.sched_rt_runtime_us = -1

ステップ 4: Namespaceの作成

oc create namespace realtime-demo

ステップ 5: リアルタイムVMのデプロイ

以下のVMマニフェストは、すべてのリアルタイムチューニングレイヤーを組み合わせています。ここでは標準的なFedoraコンテナディスクを使用します。リアルタイムカーネルを備えた専用のRTゲストイメージを使用するのが最良の結果を得られますが、標準イメージでもホスト側の構成を実証および検証するには十分です。

主要な構成フィールド：

  • cpu.model: host-passthrough — マスキングせずにすべてのホストCPU機能をゲストに公開

  • cpu.dedicatedCpuPlacement: true — vCPUを分離された物理CPUに固定

  • cpu.isolateEmulatorThread: true — QEMUエミュレータースレッド専用のCPUを予約

  • cpu.ioThreadsPolicy: auto — I/Oスレッドをエミュレータースレッドと同じCPUに配置

  • cpu.numa.guestMappingPassthrough: {} — ホストのNUMAトポロジーをゲストに渡す

  • cpu.realtime.mask: "0" — vCPU 0をSCHED_FIFOスケジューリング用に構成。vCPU 1はハウスキーピングタスク用にデフォルトスケジューラに残る

  • memory.hugepages.pageSize: 1Gi — 1Giヒュージページプールからメモリを割り当て

  • autoattachMemBalloon: false — メモリ管理割り込みを回避するためにバルーンデバイスを無効化

  • autoattachGraphicsDevice: false — 割り込みソースを減らすためにグラフィックスデバイスを無効化

この例のcloud-initパスワードはデモンストレーション専用です。本番環境では、SSHキー認証または強力で固有のパスワードを使用してください。

vm-realtime.yamlをダウンロード

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: vm-realtime
  namespace: realtime-demo
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        kubevirt.io/domain: vm-realtime
    spec:
      nodeSelector:
        node-role.kubernetes.io/worker-rt: ""
      domain:
        cpu:
          model: host-passthrough
          sockets: 1
          cores: 2
          threads: 1
          dedicatedCpuPlacement: true
          isolateEmulatorThread: true
          ioThreadsPolicy: auto
          numa:
            guestMappingPassthrough: {}
          realtime:
            mask: "0"
        memory:
          hugepages:
            pageSize: 1Gi
        resources:
          requests:
            memory: 2Gi
            cpu: 2
          limits:
            memory: 2Gi
            cpu: 2
        devices:
          autoattachSerialConsole: true
          autoattachMemBalloon: false
          autoattachGraphicsDevice: false
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            masquerade: {}
      networks:
      - name: default
        pod: {}
      terminationGracePeriodSeconds: 0
      volumes:
      - name: rootdisk
        containerDisk:
          image: quay.io/containerdisks/fedora:latest
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            password: redhat
            chpasswd:
              expire: false
EOF

VMがRunningステータスになるまで待ちます。

oc get vm vm-realtime -n realtime-demo -w

想定される出力:

NAME          AGE   STATUS    READY
vm-realtime   90s   Running   True
VMは kubevirt.io/realtime および cpumanager=true ラベルが付いたノードにのみスケジュールされます。KubeVirtはVMスペックで realtime が指定されると、これらのセレクターを自動的に追加します。VMが Scheduling 状態のままになる場合は、ターゲットノードが両方のラベルを持っていることを確認してください。

ステップ 6: ホスト上でのリアルタイムチューニングの検証

CPUピンニングの検証

VMが実行されているノードを検索します。

oc get vmi vm-realtime -n realtime-demo -o wide

そのノードでデバッグセッションを開きます。

oc debug node/<node-name>

CPUマネージャの状態を確認し、どの物理CPUが割り当てられているかを確認します。

chroot /host cat /var/lib/kubelet/cpu_manager_state

entries マップには、2つのvCPU IDと1つのエミュレータースレッド用CPU（合計3つ）を持つ compute コンテナが表示されるはずです。割り当てられたすべてのCPUは、分離された範囲（PerformanceProfileに従って2-7）内にある必要があります。

ヒュージページの検証

chroot /host grep HugePages /proc/meminfo

HugePages_Free は、VMによって消費された分（1Giヒュージページを使用する2Gi VMの場合は2ページ）だけ HugePages_Total よりも少なくなっているはずです。

guestMappingPassthrough を使用する場合、vCPUがスケジュールされている各NUMAノードでヒュージページが利用可能である必要があります。PerformanceProfileがヒュージページをNUMAノード間で均等に分散している場合、単一ノードの割り当て分を超えるメモリを要求すると、割り当てに失敗します。VMのメモリ要求は、NUMAノードごとのヒュージページ割り当て以下に抑えてください。

デバッグシェルを終了します。

exit

ステップ 7: VM内でのリアルタイム構成の検証

VMコンソールに接続します。

virtctl console vm-realtime -n realtime-demo

ユーザー fedora、パスワード redhat でログインします。

カーネルタイプの確認

uname -r

ゲストが標準カーネル（rt サフィックスなし）を実行している場合でも、vCPUのホストレベルではリアルタイムスケジューリングがアクティブになっています。ゲストRTカーネルを使用すると、ゲスト内でのプリエンプション保証が追加されるため、本番ワークロードには推奨されます。

CPU数の確認

lscpu | grep -E "^CPU\(s\)|^Socket|^Core|^Thread"

想定される出力:

CPU(s):                  2
Thread(s) per core:      1
Core(s) per socket:      2
Socket(s):               1

スケジューリングポリシーの確認（ホストの視点から）

Ctrl+] を押してゲストコンソールから切断します。

ホストレベルのリアルタイムスケジューリングは、QEMU vCPUスレッド上でKubeVirtによって構成されます。これはホストのデバッグセッションから確認できます。VMのノードでデバッグセッションを開きます。

oc debug node/<node-name>

QEMUプロセスIDを検索します。

chroot /host pgrep qemu-kvm

前のコマンドで取得したPIDを <qemu-pid> に置き換え、そのプロセスのすべてのスレッドを、スケジューリングポリシーと優先度とともにリストします。

chroot /host ps -T -o tid,cls,rtprio,comm -p <qemu-pid>

想定される出力:

    TID CLS RTPRIO COMMAND
  70898  TS      - qemu-kvm
  70900  TS      - call_rcu
  70901  TS      - TC tc-ram-node0
  70941  TS      - IO mon_iothread
  70942  FF      1 CPU 0/KVM
  70943  TS      - CPU 1/KVM

CLS 列はスケジューリングクラスを示します（SCHED_FIFO は FF、SCHED_OTHER は TS）。RTPRIO 列はリアルタイム優先度を示します。スレッド CPU 0/KVM は優先度 1 の FF を示しており、realtime.mask: "0" で指定された通りに SCHED_FIFO でスケジュールされていることが確認できます。スレッド CPU 1/KVM は TS を示しており、ハウスキーピングタスク用にはデフォルトスケジューラが維持されていることがわかります。

デバッグシェルを終了します。

exit

ステップ 8: レイテンシーの測定とベースラインの設定

適切なレイテンシーベースラインを取得するには、cyclictest が標準的なツールです。これはリアルタイムスレッドの最悪の復帰レイテンシーを測定します。

VMコンソールに接続します。

virtctl console vm-realtime -n realtime-demo

ログイン後、realtime-tests をインストールします（cyclictest を提供します）。

sudo dnf install -y realtime-tests

短いcyclictestを実行し、3スレッドのレイテンシーを60秒間測定します。

sudo cyclictest --mlockall --smp --priority=80 --interval=200 --distance=0 --duration=60

実行結果例:

# /dev/cpu_dma_latency set to 0us
policy: fifo: loadavg: 0.92 0.42 0.18 1/148 1599

Max 列はマイクロ秒単位の最悪のレイテンシーを示します。標準（非RT）ゲストカーネルの場合、ゲスト内部のスケジューリングはプリエンプションフリーではないため、数百から数千マイクロ秒の最大レイテンシーが予想されます。RTカーネルを実行する専用のゲストイメージを使用すると、最大レイテンシーは数十マイクロ秒まで低下します。

RTワーカーノードと最適化されたゲストイメージを備えた完全にチューニングされたOpenShift Virtualizationデプロイメントでは、Red Hatのベンチマークにおいて54〜55マイクロ秒の最大レイテンシーが達成されています。詳細については、Benchmarking Red Hat OpenShift Virtualization for latency-sensitive workloads を参照してください。
レイテンシーの結果は、ハードウェア、BIOS設定、ワークロードに基づいて大きく変動します。これらの数値は例示です。必ず独自のベースラインを確立し、ワークロードのレイテンシー予算と比較してください。

トラブルシューティング

VMのスケジュール失敗：リアルタイム対応ノードがない

ターゲットノードに kubevirt.io/realtime ラベルがあることを確認します。

oc get nodes -L kubevirt.io/realtime

ラベルがない場合は、RTカーネルがインストールされ、kernel.sched_rt_runtime_us が -1 であることを確認します。PerformanceProfileはこれを自動的に設定しますが、MachineConfigPoolのロールアウトが完了していない場合、設定が適用されていない可能性があります。

PerformanceProfileがスタックしている、またはノードがデグレードしている

MachineConfigPoolのステータスを確認します。

oc get mcp worker-rt
oc describe mcp worker-rt

DEGRADED が True の場合は、影響を受けるノードの machine-config-daemon ログを調査します。

oc logs -n openshift-machine-config-operator -l k8s-app=machine-config-daemon --tail=50

一般的な原因：RTカーネルRPM用のディスク容量不足、または競合するMachineConfig。

cyclictestでの高い最大レイテンシー

cyclictest で100マイクロ秒を超える最大レイテンシーが表示される場合：

  • QEMU vCPUスレッドが SCHED_FIFO で実行されていることを確認します（ステップ7を参照）。

  • 分離されたCPUがIRQを受信していないことを確認します。RTノードで以下を確認してください。

oc debug node/<node-name> -- chroot /host cat /proc/interrupts

分離されたCPU（2〜7列）は、ほとんどのIRQラインで最小またはゼロの割り込みカウントを示すはずです。

  • BIOS設定が最適化されていることを確認します。必要に応じてハイパースレッディングを無効にし、C1よりも深いCステートを無効にし、パフォーマンスモードを有効にします。

  • 最良の結果を得るには、ゲストRTカーネルを使用してください。

VMが「insufficient hugepages（ヒュージページ不足）」で失敗する

PerformanceProfileはヒュージページを割り当てますが、既存のVMやポッドがそれらを消費している可能性があります。利用可能なヒュージページを確認します。

oc get node <node-name> -o jsonpath='{.status.allocatable.hugepages-1Gi}{"\n"}'

不足している場合は、他のヒュージページを消費するワークロードを停止するか、PerformanceProfileの pages.count を増やしてください。

クリーンアップ

VMとNamespaceを削除します。

oc delete namespace realtime-demo

PerformanceProfileを削除して、RTカーネルとCPU分離を元に戻します。

oc delete performanceprofile realtime-workers

MachineConfigPoolを削除します。

oc delete mcp worker-rt

ノードから worker-rt ロールラベルを削除します。

oc label node <node-name> node-role.kubernetes.io/worker-rt-
PerformanceProfileとMachineConfigPoolを削除すると、ノードを標準カーネルに戻すための再起動がトリガーされます。ロールアウトが完了するまで worker MachineConfigPoolを監視してください。

まとめ

本チュートリアルでは、以下の方法を学びました。

  • リアルタイムワーカーノード専用のMachineConfigPoolを作成する方法

  • PerformanceProfileを適用してRTカーネルをインストールし、CPUを分離し、ヒュージページを予約する方法

  • RTノードの準備状態を確認する方法：カーネルバージョン、分離CPU、ヒュージページ割り当て、kubevirt.io/realtime ラベル

  • レイテンシー低減設定のフルスタックを備えたリアルタイムVMをデプロイする方法

  • ホスト上でCPUピンニング、ヒュージページバックアップ、SCHED_FIFOスケジューリングを検証する方法

  • cyclictest を使用してレイテンシーを測定し、ベースラインを作成する方法

  • 各構成レイヤーが特定のジッターソースを排除する役割を理解すること
