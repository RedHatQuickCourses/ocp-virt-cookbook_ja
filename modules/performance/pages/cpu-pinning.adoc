= 仮想マシン向けの専用CPU配置
:navtitle: 専用CPU配置

== 概要

本チュートリアルでは、OpenShift Virtualizationにおいて仮想マシン向けの専用CPU配置を設定する方法を解説します。デフォルトでは、仮想マシンのvCPUはホスト上の他のワークロードと物理CPU時間を共有します。専用CPU配置を使用すると、各vCPUが特定の物理CPUに固定（ピン留め）されるため、CPUスティールが排除され、スケジューリングのジッター（揺らぎ）が軽減されます。

この設定は、予測可能なレイテンシが求められるNFV/通信アプリケーション、リアルタイム処理、HPC、およびレイテンシに敏感なデータベースなどのワークロードに不可欠です。

== 前提条件

* OpenShift VirtualizationオペレーターがインストールされたOpenShift 4.18以上（4.21で検証済み）
* クラスター管理者アクセス権（`system:admin` または同等の権限）
* `oc` および `virtctl` CLIがインストール済みであること
* 十分な物理CPUコアを持つワーカーノード（少なくとも4コア推奨）
* KubernetesのQoSクラスに関する知識（xref:resource-limits-qos.adoc[]を参照）

== 専用CPU配置の理解

OpenShiftはKubernetesのCPUマネージャーを使用して、Podに排他的な物理CPUを割り当てます。CPUマネージャーが `static` ポリシーで実行されている場合、Podのコンテナーをホストの専用CPUに固定し、他のワークロードがそれらのCPUを使用できないように制限できます。

KubeVirtの `dedicatedCpuPlacement` フィールドは、CPUマネージャーに対して `virt-launcher` PodのvCPUを物理CPUに固定するように指示します。Podが実行されると、各ゲストvCPUは予約された物理CPUに直接マッピングされます。

専用配置が機能するには、以下の2つの要件を満たす必要があります。

* ノードのCPUマネージャーが `static` ポリシーで実行されていること。
* VMのQoSが *Guaranteed* であること：CPUおよびメモリの要求量（request）が上限（limit）と一致しており、かつCPU数が整数である必要があります。

[cols="1,2,2",options="header"]
|===
|モード |vCPUの動作 |ユースケース

|デフォルト（共有）
|vCPUが他のすべてのワークロードとホストCPU時間を競合する
|汎用VM、開発環境、レイテンシが重要ではないワークロード

|専用 (`dedicatedCpuPlacement: true`)
|vCPUが予約済み物理CPUに固定され、CPUスティールが発生しない
|NFV/通信、リアルタイム処理、HPC、レイテンシに敏感なデータベース
|===

== ステップ 1: CPUマネージャーのスタティックポリシーを有効にする

CPUマネージャーのポリシーは、`KubeletConfig` リソースを使用してMachineConfigPoolごとに設定します。

まず、この設定の対象を識別するために、ワーカーのMachineConfigPoolにラベルを追加します。

[source,bash,role=execute]
----
oc label machineconfigpool worker custom-kubelet=cpumanager-enabled
----

NOTE: シングルノードOpenShift (SNO) クラスターの場合、ノードは `worker` ではなく `master` MachineConfigPoolに属します。上記のコマンドおよび後述する `KubeletConfig` の `machineConfigPoolSelector` において、`worker` を `master` に置き換えてください。

xref:attachment$cpu-pinning/kubeletconfig-static-policy.yaml[kubeletconfig-static-policy.yaml をダウンロード]

`KubeletConfig` を適用して、スタティックCPUマネージャーポリシーを有効にします。

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: machineconfiguration.openshift.io/v1
kind: KubeletConfig
metadata:
  name: cpumanager-enabled
spec:
  machineConfigPoolSelector:
    matchLabels:
      custom-kubelet: cpumanager-enabled
  kubeletConfig:
    cpuManagerPolicy: static
    cpuManagerReconcilePeriod: 5s
EOF
----

Machine Config Operatorが、ターゲットプール内の全ノードに新しいkubelet設定を適用します。ロールアウトを監視します。

[source,bash,role=execute]
----
oc get mcp worker -w
----

`UPDATING` 列が `False` に戻り、`READYMACHINECOUNT` が `MACHINECOUNT` と一致するまで待ちます。

----
NAME     CONFIG                                             UPDATED   UPDATING   DEGRADED
worker   rendered-worker-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx   True      False      False
----

=== CPUマネージャーポリシーの確認

各ワーカーノードのkubeletが `static` ポリシーで実行されていることを確認します。

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host cat /etc/kubernetes/kubelet.conf
----

出力に以下が含まれている必要があります。

----
cpuManagerPolicy: static
cpuManagerReconcilePeriod: 5s
----

== ステップ 2: 専用CPUワークロード用のノードにラベルを付ける

専用CPU配置を要求するVMは、CPUマネージャーが実行されているノードのみをターゲットにするためにノードセレクターが必要です。スタティックポリシーが有効な各ワーカーノードに `cpumanager=true` ラベルを適用します。

[source,bash,role=execute]
----
oc label node <node-name> cpumanager=true
----

ラベルが付与されていることを確認します。

[source,bash,role=execute]
----
oc get nodes -L cpumanager
----

期待される出力：

----
NAME         STATUS   ROLES    AGE   VERSION   CPUMANAGER
worker-01    Ready    worker   10d   v1.34.5   true
worker-02    Ready    worker   10d   v1.34.5   true
----

NOTE: KubeVirt CRで `CPUManager` フィーチャーゲートが有効になっている場合、KubeVirtが自動的に `cpumanager=true` ラベルを管理できます。有効な場合、KubeVirtはスタティックCPUマネージャーポリシーが実行されているノードを検出し、ラベルの付与・削除を自動的に行います。フィーチャーゲートがない場合は、ノードの追加や再設定のたびにこのラベルを手動で管理してください。

== ステップ 3: 名前空間の作成

[source,bash,role=execute]
----
oc create namespace cpu-pinning-demo
----

== ステップ 4: 専用CPU配置でVMをデプロイする

以下のVM仕様は、`dedicatedCpuPlacement: true` を使用して2つの専用vCPUを要求します。`nodeSelector` により、VMは `cpumanager=true` とラベル付けされたノードでのみスケジューリングされます。メモリの要求量と上限は同一であり、Guaranteed QoS要件を満たしています。

WARNING: この例のcloud-initパスワードはデモンストレーション目的のみです。本番環境では、SSHキー認証または強力でユニークなパスワードを使用してください。

xref:attachment$cpu-pinning/vm-dedicated-cpu.yaml[vm-dedicated-cpu.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: vm-dedicated-cpu
  namespace: cpu-pinning-demo
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        kubevirt.io/domain: vm-dedicated-cpu
    spec:
      nodeSelector:
        cpumanager: "true"
      domain:
        cpu:
          sockets: 2
          cores: 1
          threads: 1
          dedicatedCpuPlacement: true
        memory:
          guest: 1Gi
        resources:
          requests:
            memory: 2Gi
          limits:
            memory: 2Gi
        devices:
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            masquerade: {}
      networks:
      - name: default
        pod: {}
      volumes:
      - name: rootdisk
        containerDisk:
          image: quay.io/containerdisks/fedora:latest
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            password: redhat
            chpasswd:
              expire: false
EOF
----

NOTE: QEMUおよびハイパーバイザーのオーバーヘッドを考慮し、メモリ制限（2Gi）はゲストメモリ（1Gi）よりも高く設定されています。KubeVirtがどのようにVMのメモリ仕様をコンテナーリソース値に変換するかについては、xref:resource-limits-qos.adoc[]を参照してください。

VMがRunning状態になるまで待ちます。

[source,bash,role=execute]
----
oc get vm vm-dedicated-cpu -n cpu-pinning-demo -w
----

期待される出力：

----
NAME              AGE   STATUS    READY
vm-dedicated-cpu  30s   Running   True
----

== ステップ 5: CPUピン留めの確認

=== ホスト側での確認

VMが実行されているノードを特定します。

[source,bash,role=execute]
----
oc get vmi vm-dedicated-cpu -n cpu-pinning-demo -o wide
----

出力例：

----
NAME              AGE   PHASE     IP            NODENAME    READY
vm-dedicated-cpu  60s   Running   10.128.0.45   worker-01   True
----

そのノードでデバッグセッションを開きます。

[source,bash,role=execute]
----
oc debug node/<node-name>
----

デバッグシェル内で、CPUマネージャーの状態ファイルを確認します。このファイルには、各コンテナーにどの物理CPUが割り当てられているかが記録されています。

[source,bash,role=execute]
----
chroot /host cat /var/lib/kubelet/cpu_manager_state
----

出力はJSONオブジェクトです。`entries` マップは `virt-launcher` PodのUIDをキーとしており、各エントリにはそのPod内の `compute` コンテナーに排他的に割り当てられた物理CPU IDがリストされています。

----
{"policyName":"static","defaultCpuSet":"0-1,3-41,43-79","entries":{"<pod-uid>":{"compute":"2,42"}},"checksum":...}
----

この例では、CPU `2` と `42` がVMの2つのvCPU専用に予約されています。これらは `defaultCpuSet` から削除されるため、VMの実行中には他のワークロードがそれらを使用することはできません。

デバッグシェルを終了します。

[source,bash,role=execute]
----
exit
----

=== ゲスト側での確認

VMコンソールに接続します。

[source,bash,role=execute]
----
virtctl console vm-dedicated-cpu -n cpu-pinning-demo
----

ユーザー `fedora`、パスワード `redhat` でログインします。ゲストOS内で、オペレーティングシステムから認識されているvCPUの数を確認します。

[source,bash,role=execute]
----
lscpu | grep -E "^CPU\(s\)|^Socket|^Core|^Thread"
----

期待される出力：

----
CPU(s):                  2
Thread(s) per core:      1
Core(s) per socket:      1
Socket(s):               2
----

ゲストは2つのvCPU（2ソケット、1コア、各1スレッド）を認識しており、要求されたトポロジーと一致しています。

`Ctrl+]` を押してコンソールから切断します。

== エミュレーター・スレッドの分離

QEMUは、I/O、タイマー、その他のハイパーバイザー・タスクを処理するエミュレーター・スレッドを実行します。デフォルトでは、このスレッドはVMのvCPUと物理CPU時間を共有するため、レイテンシのスパイクを引き起こす可能性があります。`isolateEmulatorThread` オプションは、このスレッド専用に追加の物理CPUを予約します。

エミュレーター・スレッドの分離を有効にするには、VMスペックのCPUセクションに `dedicatedCpuPlacement: true` と併せて `isolateEmulatorThread: true` を追加します。

[source,yaml]
----
domain:
  cpu:
    sockets: 2
    cores: 1
    threads: 1
    dedicatedCpuPlacement: true
    isolateEmulatorThread: true
----

この設定により、KubeVirtは合計3つの物理CPUを要求します（2つはvCPU用、1つはエミュレーター・スレッド専用）。エミュレーターに起因するわずかなジッターにも敏感なワークロードの場合にこのオプションを使用してください。

NOTE: Simultaneous Multithreading (SMT) が有効なノードで `full-pcpus-only` CPUマネージャーオプションを使用している場合、偶数のvCPUで `isolateEmulatorThread` を使用するとスケジューリングエラーが発生することがあります。これを解決するには、KubeVirt CRで `AlignCPUs` フィーチャーゲートを有効にしてください。

== トラブルシューティング

=== VMがPending状態のままになる

スケジューリングの失敗がないかVMIイベントを確認します。

[source,bash,role=execute]
----
oc describe vmi vm-dedicated-cpu -n cpu-pinning-demo
----

以下のようなイベントを探します。

----
Warning  FailedScheduling  0/3 nodes are available: 3 node(s) didn't match Pod's node affinity/selector
----

*考えられる原因と解決策:*

* *`cpumanager=true` ラベル付きノードがない*: `oc get nodes -L cpumanager` を実行し、スタティックCPUマネージャーポリシーが適用されているノードにラベルを付与します。
* *利用可能な固定可能CPUが不足している*: CPUマネージャーは割り当て可能なプールからCPUを予約します。`oc describe node <node-name>` でノードのキャパシティを確認し、要求を満たすのに十分なCPUがあるか検証します。

=== VM起動後にCPUピン留めが有効にならない

KubeletConfigが正しく適用されたか確認します。

[source,bash,role=execute]
----
oc get kubeletconfig cpumanager-enabled
----

MachineConfigPoolの状態を確認します。

[source,bash,role=execute]
----
oc get mcp worker
----

`UPDATED` が `False` または `DEGRADED` が `True` の場合、設定は完全に適用されていません。MCOログを調査してください。

[source,bash,role=execute]
----
oc logs -n openshift-machine-config-operator -l k8s-app=machine-config-operator --tail=50
----

=== VMが "insufficient dedicated CPU resources" でアドミッションに失敗する

このエラーは、`dedicatedCpuPlacement: true` が設定されているが、QoSクラスがGuaranteedではないことを意味します。CPU要求量と上限が等しいこと、および両方が整数値であることを確認してください。

virt-launcher Podに実際に割り当てられているQoSクラスを確認します。

[source,bash,role=execute]
----
oc get pod -n cpu-pinning-demo -l kubevirt.io/domain=vm-dedicated-cpu -o jsonpath='{.items[0].status.qosClass}{"\n"}'
----

値は `Guaranteed` である必要があります。`Burstable` の場合は、VMのリソース仕様を見直し、メモリ要求量とメモリ上限が一致していることを確認してください。

=== `cpu_manager_state` にVMのエントリが表示されない

CPUマネージャー状態ファイルにvirt-launcher PodのCPU割り当てがリストされていない場合、ノードのkubeletがまだ `none` ポリシーで動作している可能性があります。ポリシーを確認します。

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host cat /etc/kubernetes/kubelet.conf
----

`cpuManagerPolicy` が存在しないか `none` に設定されている場合、KubeletConfigがそのノードにまだ適用されていません。MachineConfigPoolのロールアウトが完了するまで待機してください。

== クリーンアップ

本チュートリアルで作成した全リソースを削除します。

[source,bash,role=execute]
----
oc delete namespace cpu-pinning-demo
----

必要なくなった場合は、ノードから `cpumanager=true` ラベルを削除します。

[source,bash,role=execute]
----
oc label node <node-name> cpumanager-
----

CPUマネージャーポリシーを元に戻すには、`KubeletConfig` を削除し、MachineConfigPoolからラベルを削除します。

[source,bash,role=execute]
----
oc delete kubeletconfig cpumanager-enabled
----

[source,bash,role=execute]
----
oc label machineconfigpool worker custom-kubelet-
----

WARNING: `KubeletConfig` を削除すると、kubelet設定を元に戻すために別のMCOロールアウトがトリガーされます。CPUマネージャーの動作に依存する追加のワークロードをスケジュールする前に、MachineConfigPoolのロールアウトが完了したことを確認してください。

== まとめ

このチュートリアルでは、以下の方法を学びました。

* `KubeletConfig` リソースを使用して、Kubernetes CPUマネージャーで `static` ポリシーを設定する
* 専用CPU配置が利用可能であることを示すノードラベルを付ける
* `dedicatedCpuPlacement: true` とGuaranteed QoSを持つ仮想マシンをデプロイする
* CPUマネージャー状態ファイルを使用して、ホスト側からCPUピン留めを確認する
* ゲスト内からvCPUトポロジーを確認する
* 必要に応じて `isolateEmulatorThread: true` を使用してQEMUエミュレーター・スレッドを分離する
