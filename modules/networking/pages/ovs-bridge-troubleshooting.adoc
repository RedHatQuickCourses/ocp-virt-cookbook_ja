= Localnetネットワーク用のOVSブリッジ作成のトラブルシューティング
:navtitle: Localnetネットワークのトラブルシューティング

検証済みバージョン:
----
OpenShift 4.20
----

このガイドは、OpenShift Virtualizationにおいて、Localnetセカンダリネットワーク用のOVS (Open vSwitch) ブリッジが正しく作成および構成されていることを確認するのに役立ちます。

== OVSブリッジとLinuxブリッジの理解

**重要**: OVSブリッジの動作はLinuxブリッジとは異なります。

* **OVSブリッジ**は、`ovs-vsctl`コマンドを使用して確認できます。
* **OVSブリッジ**は、`ip addr show`や`ip link show`では通常のネットワークインターフェースとして表示されない場合があります。
* これは**正常かつ想定された動作**です。OVSブリッジは標準的なLinuxネットワークデバイスではなく、OVSデータベース内に存在するためです。

== 検証プロセス

=== ステップ 1: NodeNetworkConfigurationPolicyのステータス確認

まず、NNCPが適用され、正常に構成されていることを確認します。

[source,bash,role=execute]
----
# すべてのNNCPを一覧表示
oc get nncp

# 特定のNNCPのステータスを確認
oc get nncp br-vlan-creation
----

**期待される出力:**

----
NAME               STATUS      REASON
br-vlan-creation   Available   SuccessfullyConfigured
----

=== ステップ 2: NNCPの詳細情報の取得

NNCPの完全な構成とステータスを確認します。

[source,bash,role=execute]
----
oc get nncp br-vlan-creation -o yaml
----

**出力内の以下の重要な指標に注目してください:**

[source,yaml]
----
status:
  conditions:
  - lastHeartbeatTime: "2025-10-06T20:38:57Z"
    lastTransitionTime: "2025-10-06T20:38:57Z"
    message: 1/1 nodes successfully configured
    reason: SuccessfullyConfigured
    status: "True"
    type: Available
  - reason: SuccessfullyConfigured
    status: "False"
    type: Degraded
----

* `Available: True` かつ `SuccessfullyConfigured` = 構成が適用済み
* `Degraded: False` = 問題は検出されていない
* `message: X/X nodes successfully configured` = すべてのノードが正常に構成されている

=== ステップ 3: OVSブリッジが存在することの確認

**これがOVSブリッジを確認するための正しい方法です:**

[source,bash,role=execute]
----
# ノード名を取得
oc get nodes

# すべてのOVSブリッジを一覧表示
oc debug node/<node-name> -- chroot /host ovs-vsctl list-br
----

**出力例:**

----
Starting pod/ocplab01-debug-k7bfc ...
To use host binaries, run `chroot /host`
br-ex
br-int
br-vlan

Removing debug pod ...
----

このリストに `br-vlan` が表示されていれば、ブリッジは正常に作成されています。

=== ステップ 4: ブリッジ構成の詳細確認

完全なOVS構成を確認して、ブリッジポートとインターフェースを表示します。

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host ovs-vsctl show
----

**出力例 (関連部分):**

----
Bridge br-vlan
    Port enp11s0f0np0
        Interface enp11s0f0np0
            type: system
----

**確認項目:**

* ブリッジ名 (`br-vlan`) が出力にあること
* 物理インターフェース (例: `enp11s0f0np0`) がポートとしてリストされていること
* 物理インターフェースのインターフェースタイプが `system` であること

=== ステップ 5: 物理インターフェースのステータス確認

物理インターフェースがブリッジに接続されており、UP状態であることを確認します。

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host ip link show <interface-name>
----

**出力例:**

----
4: enp11s0f0np0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc mq master ovs-system state UP mode DEFAULT group default qlen 1000
    link/ether e8:eb:d3:13:06:16 brd ff:ff:ff:ff:ff:ff
----

**確認項目:**

* `UP,LOWER_UP` - インターフェースが動作可能であること
* `master ovs-system` - インターフェースがOVSによって管理されていること
* `state UP` - インターフェースの状態がアップしていること

=== ステップ 6: なぜ "ip addr show br-vlan" が失敗するのか

**これは正常な動作です:**

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host ip addr show br-vlan
----

**期待される結果:**

----
Device "br-vlan" does not exist.
----

**この理由:**

* OVSブリッジは通常のLinuxネットワークデバイスとしてではなく、OVSデータベース内に存在するためです。
* 内部インターフェースを持たない限り、標準の `ip` コマンド出力には表示されません。
* これはブリッジの作成失敗を意味するものではありません。
* OVSブリッジを確認する際は、必ず `ovs-vsctl` コマンドを使用してください。

== よくある問題と解決策

=== 問題 1: ovs-vsctl list-br にブリッジが表示されない

**症状:**

* NNCPは `Available: True` を示しているが、ブリッジが `ovs-vsctl list-br` に表示されない

**トラブルシューティング:**

[source,bash,role=execute]
----
# NNCPの条件を確認
oc get nncp br-vlan-creation -o jsonpath='{.status.conditions[?(@.type=="Available")]}' | jq

# NMStateオペレーターのログを確認
oc logs -n openshift-nmstate -l app=kubernetes-nmstate-operator

# 問題のあるノードネットワーク状態があるか確認
oc get nns -A
----

**一般的な原因:**

* 物理インターフェース名の不一致 (`ip link show` で確認)
* ノード上でOVSが正しく実行されていない
* NMStateオペレーターの問題

=== 問題 2: 物理インターフェースがブリッジに接続されていない

**症状:**

* ブリッジは存在するが、`ovs-vsctl show` でインターフェースがポートとして表示されない

**トラブルシューティング:**

[source,bash,role=execute]
----
# ブリッジ上のすべてのポートを一覧表示
oc debug node/<node-name> -- chroot /host ovs-vsctl list-ports br-vlan

# インターフェースのステータスを確認
oc debug node/<node-name> -- chroot /host ip link show <interface-name>

# NNCP構成に正しいインターフェース名があるか確認
oc get nncp br-vlan-creation -o yaml | grep -A 5 "name: <interface-name>"
----

**解決策:**

. NNCP内の物理インターフェース名が正しいことを確認します。
. インターフェースが別のブリッジによって既に使用されていないか確認します。
. インターフェースが存在し、ノードで使用可能であることを確認します。

=== 問題 3: ノードセレクターの不一致

**症状:**

* NNCPは成功しているが、ブリッジが存在しない

**トラブルシューティング:**

[source,bash,role=execute]
----
# ノードラベルを確認
oc get nodes --show-labels

# NNCPノードセレクターを確認
oc get nncp br-vlan-creation -o jsonpath='{.spec.nodeSelector}'
----

**一般的な問題:**

* シングルノードOpenShift (SNO) ノードには `control-plane,master,worker` のロールがある
* `node-role.kubernetes.io/worker: ""` セレクターを持つNNCPは引き続き一致する
* NNCPステータスが "X/X nodes successfully configured" を示しているか確認してください

=== 問題 4: ブリッジは作成されたが機能していない

**症状:**

* ブリッジは存在し、インターフェースも接続されているが、トラフィックが流れない

**トラブルシューティング:**

[source,bash,role=execute]
----
# インターフェースのキャリアステータスを確認
oc debug node/<node-name> -- chroot /host cat /sys/class/net/<interface-name>/carrier

# インターフェースエラーを確認
oc debug node/<node-name> -- chroot /host ethtool -S <interface-name>

# 物理リンクがアップしているか確認
oc debug node/<node-name> -- chroot /host ethtool <interface-name> | grep "Link detected"
----

**一般的な原因:**

* 物理ケーブルが接続されていない
* スイッチポートが無効になっている、または構成が間違っている
* インターフェースの速度/デュプレックスの不一致

== Localnet用ブリッジマッピングの確認

OVSブリッジを作成した後、OVN-Kubernetesへのブリッジマッピングが正しく機能していることを確認する必要があります。

=== ブリッジマッピングとは何か

ブリッジマッピングは、**論理ネットワーク名** (NetworkAttachmentDefinitionsで使用) を**物理OVSブリッジ**に接続します。例えば:

* 論理名: `localnet-vlan`
* 物理ブリッジ: `br-vlan`
* マッピング: `localnet-vlan:br-vlan`

=== ステップ 1: ブリッジマッピングのNNCPステータス確認

ブリッジマッピングのNNCPが正常に適用されたか確認します:

[source,bash,role=execute]
----
# すべてのNNCPを一覧表示
oc get nncp | grep mapping

# 特定のブリッジマッピングNNCPを確認
oc get nncp localnet-vlan-bridge-mapping
----

**期待される出力:**

----
localnet-vlan-bridge-mapping   Available   SuccessfullyConfigured
----

=== ステップ 2: ブリッジマッピングの詳細構成を取得

[source,bash,role=execute]
----
oc get nncp localnet-vlan-bridge-mapping -o yaml
----

**出力内の以下の重要なセクションに注目してください:**

[source,yaml]
----
spec:
  desiredState:
    ovn:
      bridge-mappings:
      - bridge: br-vlan
        localnet: localnet-vlan
        state: present
  nodeSelector:
    node-role.kubernetes.io/worker: ""
status:
  conditions:
  - message: 1/1 nodes successfully configured
    reason: SuccessfullyConfigured
    status: "True"
    type: Available
----

=== ステップ 3: ノード上のOVNブリッジマッピングを確認

これが**非常に重要な検証作業**です。OVN-Kubernetesにブリッジマッピングが構成されているか確認します:

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host ovs-vsctl get Open_vSwitch . external_ids:ovn-bridge-mappings
----

**期待される出力:**

----
"localnet-vlan:br-vlan,physnet:br-ex"
----

**この意味:**

* `localnet-vlan:br-vlan` = カスタムマッピング
* `physnet:br-ex` = デフォルトのOpenShiftマッピング
* 複数のマッピングがある場合はカンマで区切られます

=== ステップ 4: 完全なOVS外部IDの確認

すべてのOVS外部IDを取得して、完全な構成を確認します:

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host ovs-vsctl list Open_vSwitch | grep -A 2 external_ids
----

**出力例:**

----
external_ids        : {hostname=ocplab01, ovn-bridge-mappings="localnet-vlan:br-vlan,physnet:br-ex", ovn-bridge-remote-probe-interval="0", ovn-enable-lflow-cache="true", ovn-encap-ip="192.168.2.241", ovn-encap-type=geneve, ovn-is-interconn="true", ovn-memlimit-lflow-cache-kb="1048576", ovn-monitor-all="true", ovn-ofctrl-wait-before-clear="0", ovn-remote="unix:/var/run/ovn/ovnsb_db.sock", ovn-remote-probe-interval="180000", ovn-set-local-ip="true", rundir="/var/run/openvswitch", system-id="1b41fc3f-f597-4cd9-b508-e8247c209d97"}
----

**確認事項:**

* `ovn-bridge-mappings` にマッピングが含まれていること
* `hostname` がノードと一致すること
* `ovn-encap-type=geneve` が存在すること

=== ステップ 5: 物理インターフェースの接続確認

物理インターフェースがブリッジに接続されていることを確認します:

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host ovs-vsctl list-ports br-vlan
----

**期待される出力:**

----
enp11s0f0np0
----

これで、物理インターフェースがブリッジ上のポートであり、トラフィックを送信できる状態であることが確認できます。

== ブリッジマッピングのトラフィックフロー

ブリッジマッピングを介したトラフィックフローの理解:

│   └─> eth1 (セカンダリネットワークインターフェース)                │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ NetworkAttachmentDefinition                                      │
│   - name: localnet-vlan-100                                      │
│   - topology: localnet                                           │
│   - vlanID: 100                                                  │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ OVN-Kubernetes (ブリッジマッピングを参照)                         │
│   localnet-vlan → br-vlan                                        │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ OVS Bridge: br-vlan                                              │
│   - VLANタグ 100 を追加                                           │
│   - 物理インターフェースへルーティング                             │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ Physical Interface: enp11s0f0np0                                 │
│   - VLANタグ付きトラフィックを物理ネットワークへ送信               │
└─────────────────────────────────────────────────────────────────┘
----

== ブリッジマッピングの問題のトラブルシューティング

=== 問題 1: ブリッジマッピングがOVNの外部IDにない

**症状:**

* NNCPは成功しているが、`ovs-vsctl get Open_vSwitch . external_ids:ovn-bridge-mappings` にマッピングが含まれていない

**トラブルシューティング:**

[source,bash,role=execute]
----
# NNCPが適用されたか確認
oc get nncp localnet-vlan-bridge-mapping -o yaml

# NMStateオペレーターを再起動
oc delete pod -n openshift-nmstate -l app=kubernetes-nmstate-operator

# NMStateハンドラーのログを確認
oc logs -n openshift-nmstate -l component=kubernetes-nmstate-handler
----

**一般的な原因:**

* ブリッジが作成される前にNNCPが適用された
* NMStateハンドラーが正常に動作していない
* OVSサービスの不具合

**解決策:**

ブリッジが存在することを確認した後、ブリッジマッピングのNNCPを削除して再作成してください。

=== 問題 2: マッピング内のブリッジ名が間違っている

**症状:**

* ブリッジマッピングは存在するが、間違ったブリッジ名を使用している

**検証:**

[source,bash,role=execute]
----
# 存在するブリッジを確認
oc debug node/<node-name> -- chroot /host ovs-vsctl list-br

# マッピングが参照しているものを確認
oc get nncp localnet-vlan-bridge-mapping -o jsonpath='{.spec.desiredState.ovn.bridge-mappings[0].bridge}'
----

**解決策:**

マッピングNNCP内のブリッジ名が、実際のOVSブリッジ名と完全に一致していることを確認してください。

=== 問題 3: 論理名の不一致

**症状:**

* ブリッジマッピングが存在するにもかかわらず、VMがネットワークに接続できない

**検証:**

[source,bash,role=execute]
----
# ブリッジマッピング内の論理名を確認
oc get nncp localnet-vlan-bridge-mapping -o jsonpath='{.spec.desiredState.ovn.bridge-mappings[0].localnet}'

# NetworkAttachmentDefinition または ClusterUserDefinedNetwork (CUDN) が同じ名前を参照しているか確認
oc get network-attachment-definitions <nad-name> -n <namespace> -o yaml
----

**解決策:**

ブリッジマッピング内の論理ネットワーク名は、NetworkAttachmentDefinition または ClusterUserDefinedNetwork 構成で使用されている名前と一致する必要があります。

=== 問題 4: 複数のマッピングの競合

**症状:**

* 複数のブリッジマッピングが構成されているが、トラフィックが間違ったブリッジに送信される

**検証:**

[source,bash,role=execute]
----
# すべてのブリッジマッピングを一覧表示
oc debug node/<node-name> -- chroot /host ovs-vsctl get Open_vSwitch . external_ids:ovn-bridge-mappings
----

**期待される形式:**

----
"mapping1:bridge1,mapping2:bridge2,mapping3:bridge3"
----

**解決策:**

各論理ネットワーク名がユニークであり、正しいブリッジにマッピングされていることを確認してください。OVNは最初の一致するマッピングを使用します。

== 完全な検証チェックリスト

このチェックリストを使用して、OVSブリッジとブリッジマッピングが正しく構成されているかを確認してください:

=== ブリッジ作成

* [ ] NNCPステータスが `Available: True` かつ `SuccessfullyConfigured` になっていること
* [ ] NNCPメッセージがすべてのノードで正常に構成されたことを示していること
* [ ] `ovs-vsctl list-br` にブリッジ名が含まれていること
* [ ] `ovs-vsctl show` で物理インターフェースがポートとしてブリッジに表示されていること
* [ ] 物理インターフェースが `UP,LOWER_UP` ステータスであること
* [ ] 物理インターフェースが `master ovs-system` であること
* [ ] 物理リンクがキャリア状態であること (ケーブル接続済み)
* [ ] NMStateオペレーターログにエラーがないこと

=== ブリッジマッピング

* [ ] ブリッジマッピングNNCPが `Available: True` であること
* [ ] `ovs-vsctl get Open_vSwitch . external_ids:ovn-bridge-mappings` にマッピングが含まれていること
* [ ] マッピング形式が正しいこと: `logical-name:bridge-name`
* [ ] 論理ネットワーク名が NetworkAttachmentDefinition (NAD) / CUDN で使用するものと一致していること
* [ ] マッピング内のブリッジ名が実際のOVSブリッジと一致していること
* [ ] 物理インターフェースがブリッジに接続されていること (`ovs-vsctl list-ports`)
* [ ] 他のブリッジマッピングとの競合がないこと

== ClusterUserDefinedNetwork (CUDN) の検証

NetworkAttachmentDefinitionの代わりにClusterUserDefinedNetworkを使用する場合、追加の検証手順が必要です。

=== ステップ 1: CUDNリソースの確認

ClusterUserDefinedNetworkが正常に作成されたか確認します:

[source,bash,role=execute]
----
# すべてのCUDNを一覧表示
oc get clusteruserdefinednetwork

# 詳細なCUDN情報を取得
oc get clusteruserdefinednetwork <cudn-name> -o yaml
----

**期待される出力:**

----
NAME                     AGE
cudn-localnet-vlan-100   5m
----

=== ステップ 2: CUDN構成の確認

CUDN specを確認して、適切な構成であることを確かめます:

[source,bash,role=execute]
----
oc get clusteruserdefinednetwork cudn-localnet-vlan-100 -o yaml
----

**確認すべき重要なフィールド:**

[source,yaml]
----
spec:
  namespaceSelector: 
    matchExpressions: 
    - key: kubernetes.io/metadata.name
      operator: In 
      values: ["vm-guests-vlan-prod", "vm-guests-vlan-dev"]
  network:
    topology: Localnet 
    localnet:
      role: Secondary 
      physicalNetworkName: localnet-vlan  # ブリッジマッピングの論理名と一致すること
      vlanID: 100
      ipam:
        mode: Disabled  # ニーズに応じてStaticなども可
----

=== ステップ 3: 自動作成されたNetworkAttachmentDefinitionの確認

CUDNは選択された名前空間内に自動的にNetworkAttachmentDefinitionを作成します:

[source,bash,role=execute]
----
# すべての名前空間のNADを一覧表示
oc get network-attachment-definitions -A | grep <cudn-name>

# 特定の名前空間を確認
oc get network-attachment-definitions -n <namespace>
----

**期待される出力:**

----
NAMESPACE             NAME                     AGE
vm-guests-vlan-dev    cudn-localnet-vlan-100   5m
vm-guests-vlan-prod   cudn-localnet-vlan-100   5m
----

=== ステップ 4: physicalNetworkName がブリッジマッピングと一致することを確認

CUDN内の `physicalNetworkName` は、ブリッジマッピング内の論理ネットワーク名と一致する必要があります:

[source,bash,role=execute]
----
# CUDNのphysicalNetworkNameを取得
oc get clusteruserdefinednetwork <cudn-name> -o jsonpath='{.spec.network.localnet.physicalNetworkName}'

# ブリッジマッピングの論理名を取得
oc get nncp <bridge-mapping-nncp> -o jsonpath='{.spec.desiredState.ovn.bridge-mappings[0].localnet}'

# OVN内で一致することを確認
oc debug node/<node-name> -- chroot /host ovs-vsctl get Open_vSwitch . external_ids:ovn-bridge-mappings
----

**すべてが同じ名前 (例: `localnet-vlan`) を示す必要があります。**

=== ステップ 5: 名前空間を跨いだネットワーク可用性の確認

異なる名前空間のVMが両方ともCUDNを使用できるかテストします:

[source,bash,role=execute]
----
# 最初の名前空間のVMを確認
oc get vmi -n vm-guests-vlan-prod -o jsonpath='{.items[*].status.interfaces[?(@.name=="cudn_vlan_network")].ipAddress}'

# 2番目の名前空間のVMを確認
oc get vmi -n vm-guests-vlan-dev -o jsonpath='{.items[*].status.interfaces[?(@.name=="cudn_vlan_network")].ipAddress}'
----

VMがCUDNネットワークに接続されていれば、両方ともIPアドレスが返されるはずです。

=== CUDN固有のよくある問題

. **選択した名前空間にNADが作成されない**:
   * **症状**: CUDNは存在するが、`oc get network-attachment-definitions -n <namespace>` に何も表示されない
   * **原因**: 名前空間セレクターが名前空間ラベルと一致しない
   * **解決策**: 名前空間ラベルがCUDNセレクターと一致することを確認する
+
[source,bash,role=execute]
----
# 名前空間ラベルを確認
oc get namespace <namespace-name> -o jsonpath='{.metadata.labels}'

# CUDNセレクターを確認
oc get clusteruserdefinednetwork <cudn-name> -o jsonpath='{.spec.namespaceSelector}'
----

. **VMが "failed bridge mapping validation" で失敗する**:
   * **症状**: CUDNが存在するにもかかわらず、VMポッドが失敗する
   * **原因**: CUDN内の `physicalNetworkName` がブリッジマッピングの論理名と一致しない
   * **解決策**: 完全一致 (大文字小文字を区別) することを確認する
+
[source,bash,role=execute]
----
# 名前の比較
oc get clusteruserdefinednetwork <cudn-name> -o jsonpath='{.spec.network.localnet.physicalNetworkName}'
oc get nncp <bridge-mapping-nncp> -o jsonpath='{.spec.desiredState.ovn.bridge-mappings[0].localnet}'
----

. **IPAMモードの問題**:
   * **症状**: VMが期待通りのIPアドレスを取得できない
   * **原因**: IPAMモードの構成ミス、またはcloud-initが機能していない
   * **cloud-init経由の固定IP**: CUDNで `ipam.mode: Disabled` に設定
   * **OVN IPAMの場合**: `staticIPAMConfig` を含めて `ipam.mode: Static` に設定
   * **検証**:
+
[source,bash,role=execute]
----
# IPAMモードを確認
oc get clusteruserdefinednetwork <cudn-name> -o jsonpath='{.spec.network.localnet.ipam.mode}'

# VM内のcloud-initログを確認
virtctl console -n <namespace> <vm-name>
# VM内: cat /var/log/cloud-init.log
----

== Localnet VLANネットワークのトラブルシューティング

=== よくある問題

. **VLANトラフィックがVMに到達しない**:
   * 物理スイッチの構成がVLANタグ付けをサポートしているか確認
   * ワーカーノード上のブリッジマッピング構成を確認
   * VLAN IDがネットワークインフラと一致しているか確認
   * 物理NICが接続されており、UP状態であることを確認
   * OVSブリッジが物理インターフェースをポートとして含んでいるか確認

. **IPアドレスの割り当て問題**:
   * DHCPサーバーがVLANネットワーク上で使用可能か確認

. **ネットワーク接続の問題**:
   * 物理ネットワークが構成済みのVLANをサポートしているか確認
   * 物理ネットワークからVLANサブネットへの接続をテスト
   * 物理スイッチポートがトランクモードになっているか確認

. **カスタムブリッジ作成の問題**:
   * 注意: OVSブリッジは `ip addr show` に表示されません。代わりに `ovs-vsctl list-br` を使用してください。
   * 物理インターフェース名が正しいことを確認してください。
   * NMStateオペレーターが正常に動作しているか確認してください。

. **ブリッジマッピングが機能しない**:
   * マッピング内のブリッジ名が作成済みのブリッジと一致しているか確認
   * OVNブリッジマッピングを確認: `ovs-vsctl get Open_vSwitch . external_ids:ovn-bridge-mappings`
   * マッピング内のlocalnet名がNAD構成と一致しているか確認

. **VMポッドが "failed bridge mapping validation" エラーで失敗する**:
   * **症状**: VMポッドが次のエラーで失敗する: `failed to find OVN bridge-mapping for network: "localnet-vlan-100"`
   * **原因**: NetworkAttachmentDefinitionに `physicalNetworkName` パラメーターがない
   * **解決策**: NADがブリッジマッピングの論理名と一致する `"physicalNetworkName": "localnet-vlan"` を含んでいるか確認する
   * **検証**: NADを更新した後、`oc delete` と `oc apply` で再作成する

== まとめ

このガイドでは、以下のことを学びました:

* OVSブリッジとLinuxブリッジの違い、およびなぜそれらが `ip addr show` 出力に表示されないのか
