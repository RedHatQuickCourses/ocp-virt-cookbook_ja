= OpenShift VirtualizationにおけるClusterUserDefinedNetworkを使用したVLAN付きLocalnetセカンダリネットワークの構成
:navtitle: Localnet VLANとCUDN

本チュートリアルでは、OpenShift Virtualizationにおいて、ClusterUserDefinedNetwork (CUDN) を使用してVLANタグ付けを行うLocalnetセカンダリネットワークを構成する方法を解説します。このアプローチでは、物理ネットワークインフラストラクチャへの直接アクセスと、複数のネームスペースにまたがるVLANセグメンテーションを提供し、Podネットワークの接続性を維持したまま、仮想マシン (VM) が特定のVLANタグ付きネットワークと通信できるようにします。CUDNは一元的な管理とネームスペース間のネットワーク可用性を提供するため、一貫したVLANベースのネットワークインフラストラクチャを必要とするエンタープライズ環境に最適です。

== 前提条件

* **NMState Operator**: クラスター内にインストールされ、実行されていること
* **NMState CR**: Operatorが実行された後に、nmstateインスタンスが作成されていること
* **ワーカーノード**: ブリッジマッピングはワーカーノードに適用されること
* **VLANインフラストラクチャ**: 物理ネットワークインフラストラクチャがVLANタグ付けをサポートしていること
* **スイッチ構成**: OpenShiftワーカーノードに接続された物理スイッチポートが、複数のVLANタグ付きトラフィックストリームを処理するために**トランクモード**で構成されていること

検証済みバージョン:
----
OpenShift 4.19
----

== ネットワーク構成要件

=== 物理スイッチ構成: トランクモード (TRUNK MODE)

Localnet VLAN構成が正しく機能するためには、OpenShiftワーカーノードに接続された物理スイッチポートを**トランクモードで構成する必要があります**:

* **トランクモード**: スイッチポートが複数のVLANのトラフィックを伝送できるようにする
* **VLANタグ付け**: 各VLANは一意のVLAN ID (例: 100, 200, 300) によって識別される
* **マルチネットワーク**: 複数のVLANベースのセカンダリネットワークをサポートするために不可欠

== ステップ 1: VMネームスペースの作成

VLAN構成のLocalnetセカンダリネットワークを使用するVM用のネームスペースを作成します。ClusterUserDefinedNetwork (CUDN) を使用すると、同一のネットワークを複数のネームスペースで利用可能にできます。

[source,bash,role=execute]
----
# Create first namespace
oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: vm-guests-vlan-prod
EOF

# Create second namespace
oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: vm-guests-vlan-dev
EOF
----

== ステップ 2: 追加NIC上へのカスタムOVSブリッジの作成

まず、プライマリクラスターネットワークからVLANトラフィックを分離するために、追加の物理ネットワークインターフェース (NIC) 上にカスタムOVSブリッジを作成します。

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: br-vlan-creation
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ''
  desiredState:
    interfaces:
    - name: enp11s0f0np0  # Replace with your physical interface name
      type: ethernet
      state: up
      ipv4:
        enabled: false
      ipv6:
        enabled: false
      description: "Physical interface for VLAN localnet"
    - name: br-vlan
      type: ovs-bridge
      state: up
      bridge:
        port:
        - name: enp11s0f0np0
      description: "OVS bridge for VLAN traffic"
EOF
----

注記: `enp11s0f0np0` は、使用する追加の物理インターフェース名に置き換えてください。利用可能なインターフェースは、以下のコマンドを実行して確認できます。

[source,bash,role=execute]
----
oc debug node/<node-name> -- ip link show
----

ブリッジの作成を確認します。

[source,bash,role=execute]
----
oc get nncp br-vlan-creation
# Expected output: STATUS: Available
----

== ステップ 3: ブリッジマッピングの構成

論理ネットワークインターフェース名 `localnet-vlan` をワーカーノード上の `br-vlan` OVSブリッジに接続するブリッジマッピングを構成します。このマッピングにより、どの物理ブリッジインターフェースがVLANサポート付きのLocalnetトラフィックを処理するかをOVN-Kubernetesに伝えます。

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: cudn-localnet-vlan-bridge-mapping
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ''  
  desiredState:
    ovn:
      bridge-mappings:
      - localnet: localnet-vlan
        bridge: br-vlan
        state: present
EOF
----

== ステップ 4: VLAN付きClusterUserDefinedNetworkの作成

複数のネームスペースにまたがるセカンダリネットワーク接続のために、VLANタグ付けを伴うLocalnetトポロジーを構成するClusterUserDefinedNetworkを作成します。

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: k8s.ovn.org/v1
kind: ClusterUserDefinedNetwork
metadata:
  name: cudn-localnet-vlan-100
spec:
  namespaceSelector: 
    matchExpressions: 
    - key: kubernetes.io/metadata.name
      operator: In 
      values: ["vm-guests-vlan-prod", "vm-guests-vlan-dev"]
  network:
    topology: Localnet 
    localnet:
      role: Secondary 
      physicalNetworkName: localnet-vlan
      vlanID: 100
      ipam:
        mode: Disabled
EOF
----

**構成の詳細**:

* `vlanID: 100`: ネットワークセグメンテーション用にVLANタグ100を指定
* `physicalNetworkName`: ブリッジマッピングの論理ネットワークインターフェース名 (`localnet-vlan`) を参照
* `role: Secondary`: これをセカンダリネットワークとして定義 (VMはPodネットワークをプライマリとして保持)
* `namespaceSelector`: このネットワークを使用できるネームスペースを定義
* **IPAM Disabled**: IPアドレスはcloud-init (静的) または外部DHCPサーバーを介して構成する必要がある

== ステップ 5: VLANネットワーク接続と静的IPを持つVMのデプロイ

構成済みのネームスペースの1つに、PodネットワークとVLANタグ付きLocalnetセカンダリネットワークの両方に接続するVMを作成します。

警告: 本例のcloud-initパスワードはデモ目的のみです。本番環境では必ず強力で一意のパスワードを使用するか、SSH鍵認証を使用してください。

[source,yaml]
[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-cudn-vlan-vm
  namespace: vm-guests-vlan-prod
  labels:
    app: fedora-cudn-vlan-vm
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-cudn-vlan-volume
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 33Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        app: fedora-cudn-vlan-vm
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            bridge: {}
          - name: cudn_vlan_network
            bridge: {}
        resources:
          requests:
            memory: 2Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      - name: cudn_vlan_network
        multus:
          networkName: cudn-localnet-vlan-100
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: fedora-cudn-vlan-volume
      - name: cloudinitdisk
        cloudInitNoCloud:
          networkData: |
            version: 2
            ethernets:
              enp2s0:
                addresses: ["192.168.100.10/24"]
          userData: |
            #cloud-config
            user: fedora
            password: fedora
            chpasswd: { expire: False }
            packages:
            - python3
            - tcpdump
            - net-tools
            write_files:
            - path: /root/index.html
              content: |
                <h1>Welcome to OpenShift Virtualization CUDN VLAN Network!</h1>
                <p>VLAN ID: 100</p>
                <p>Static IP: 192.168.100.10/24</p>
                <p>Network: CUDN Localnet</p>
                <p>Namespace: vm-guests-vlan-prod</p>
              permissions: '0644'
            runcmd:
            - nohup python3 -m http.server 8080 --directory /root > /dev/null 2>&1 &
EOF
----

== ステップ 7: VLANネットワーク接続の確認

VMコンソールにアクセスし、ネームスペース間でのVLAN構成を含むデュアルネットワーク接続を確認します。

[source,bash,role=execute]
----
# Check VMs in production namespace
oc get vm -n vm-guests-vlan-prod
oc get vmi -n vm-guests-vlan-prod

# Check VMs in development namespace
oc get vm -n vm-guests-vlan-dev
oc get vmi -n vm-guests-vlan-dev

# Access production VM console (credentials: fedora/fedora via cloud-init)
virtctl console -n vm-guests-vlan-prod fedora-cudn-vlan-vm

# Access development VM console (credentials: fedora/fedora via cloud-init)
virtctl console -n vm-guests-vlan-dev fedora-cudn-vlan-vm-dev
----

各VMコンソール内で、VLANネットワーク構成を確認します。

[source,bash]
----
# Check network interfaces - primary (pod network) and secondary (VLAN network)
ip addr show

# Check routes
ip route show

# Expected output example (primary network via DHCP):
default via 10.128.0.1 dev enp1s0 proto dhcp src 10.128.0.174 metric 100 

# Test external connectivity
ping 8.8.8.8

# Verify the web service is running
curl localhost:8080

# Verify static IP assignment on VLAN interface
ip addr show enp2s0
# Expected: inet 192.168.100.10/24 (prod) or 192.168.100.20/24 (dev)

# Test VLAN interface connectivity between VMs
# From production VM (192.168.100.10) ping development VM:
ping 192.168.100.20

# Verify VLAN tagging (if tcpdump is available)
# This will show VLAN-tagged traffic on the secondary interface
sudo tcpdump -i enp2s0 -nn vlan 100
----

== ステップ 8: ネームスペース間接続のテスト

異なるネームスペースにあるVMが、共有されたCUDN VLANネットワーク上で通信できることを確認します。

[source,bash]
----
# From production VM (192.168.100.10), test connectivity to development VM
ping 192.168.100.20

# Test web service connectivity between namespaces
curl 192.168.100.20:8080

# From development VM (192.168.100.20), test connectivity to production VM
ping 192.168.100.10
curl 192.168.100.10:8080
----

== トラブルシューティング

OVSブリッジおよびブリッジマッピングのトラブルシューティング（CUDN固有の確認手順を含む）については、xref:ovs-bridge-troubleshooting.adoc#_clusteruserdefinednetwork_cudn_verification[OVSブリッジ検証トラブルシューティングガイド]を参照してください。

=== 一般的な問題

==== VLANトラフィックがVMに到達しない

* 物理スイッチの構成がVLANタグ付けをサポートしているか確認
* ワーカーノード上のブリッジマッピング構成を確認
* VLAN IDがネットワークインフラストラクチャと一致していることを確認
* カスタムOVSブリッジ (`br-vlan`) が作成されているか確認: `oc get nncp br-vlan-creation`
* 物理インターフェースがブリッジに接続されているか確認

==== IPアドレス割り当ての問題

* cloud-initのnetworkDataで静的IPが正しく構成されているか確認
* インターフェース名が一致しているか確認 (2番目のインターフェースにはenp2s0など)
* VM内のcloud-initログを確認: `cat /var/log/cloud-init.log`

==== ネットワーク接続の問題

* 物理ネットワークが構成されたVLANをサポートしているか確認
* 物理ネットワークからVLANサブネットへの接続性をテスト
* 物理スイッチポートがトランクモードであることを確認

==== ネームスペース間の問題

* CUDN構成のネームスペースセレクターを確認
* 両方のネームスペースがvalues配列にリストされているか確認
* ターゲットネームスペースでNADが自動作成されているか確認

==== CUDN固有の問題

* CUDNの `physicalNetworkName` がブリッジマッピングの論理名 (`localnet-vlan`) と一致しているか確認
* 選択されたネームスペースでCUDNによって作成されたNADを確認: `oc get network-attachment-definitions -A | grep cudn-localnet-vlan-100`
* IPAMモードが正しく設定されているか確認 (cloud-init経由の静的IPの場合は `Disabled`)

=== 確認コマンド

[source,bash,role=execute]
----
# Check ClusterUserDefinedNetwork status
oc get clusteruserdefinednetwork

# Verify bridge mapping on worker nodes
oc get nncp cudn-localnet-vlan-bridge-mapping -o yaml

# Check VM network status in both namespaces
oc get vmi -n vm-guests-vlan-prod -o yaml | grep -A 10 networks
oc get vmi -n vm-guests-vlan-dev -o yaml | grep -A 10 networks

# View OVN-Kubernetes logs for troubleshooting
oc logs -n openshift-ovn-kubernetes -l app=ovnkube-node

# Check CUDN network attachment across namespaces
oc get network-attachment-definitions -A | grep cudn-localnet-vlan-100
----

== CUDNとNetworkAttachmentDefinitionの比較

|===
| 機能 | CUDN | NetworkAttachmentDefinition

| **スコープ**
| クラスター全体、マルチネームスペース
| ネームスペース固有

| **管理**
| 一元的な構成
| ネームスペースごとの構成

| **VLANサポート**
| ネイティブなvlanIDパラメーター
| ネイティブなvlanIDパラメーター

| **IP管理**
| 静的/Disabledモード
| IPAM経由の静的/DHCP

| **ユースケース**
| エンタープライズ、一貫したネットワーク
| アプリケーション固有のネットワーク
|===

== まとめ

本チュートリアルでは、以下の内容を学習しました。

* 追加の物理NIC上にカスタムOVSブリッジを作成し、VLANトラフィック用のブリッジマッピングを構成する方法
* 複数のネームスペースにまたがるVLANタグ付きの `ClusterUserDefinedNetwork` を作成する方法
* CUDN VLANセカンダリネットワーク上で、cloud-initを使用して静的IPアドレスを持つVMをデプロイする方法
* 同じCUDN VLANネットワークを共有するVM間で、ネームスペース間の接続性を確認する方法
* VLANベースのネットワークにおける、CUDNとNetworkAttachmentDefinitionのアプローチの違い
