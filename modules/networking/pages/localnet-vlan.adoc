= OpenShift VirtualizationにおけるNetworkAttachmentDefinitionを使用したVLAN付きlocalnetセカンダリネットワークの構成
:navtitle: NADを使用したVLAN付きlocalnetセカンダリネットワーク

本チュートリアルでは、OpenShift Virtualizationにおいて、NetworkAttachmentDefinition (NAD) を使用してVLANタグ付けを行うlocalnetセカンダリネットワークを構成する方法を解説します。このアプローチでは、VLANセグメンテーションを伴う物理ネットワークインフラストラクチャへの直接アクセスが提供され、Podネットワークの接続性を維持したまま、仮想マシン（VM）が特定のVLANタグ付きネットワークと通信できるようになります。これは、既存のVLANベースのネットワークインフラストラクチャと統合する場合や、ネットワークセグメンテーションが必要な場合に最適です。

== 前提条件

* **NMState Operator**: クラスター内にインストールされ、実行中であること
* **NMState CR**: Operatorが実行された後にnmstateインスタンスが作成されていること
* **ワーカーノード**: ブリッジマッピングがワーカーノードに適用されること
* **VLANインフラストラクチャ**: 物理ネットワークインフラストラクチャがVLANタグ付けをサポートしていること
* **スイッチ設定**: OpenShiftワーカーノードに接続されている物理スイッチポートが、複数のVLANタグ付きトラフィックストリームを処理するために**トランクモード**で構成されていること

検証済みバージョン:
----
OpenShift 4.19
----

== ネットワーク構成要件

=== 物理スイッチの設定: トランクモード

OpenShiftワーカーノードに接続されている物理スイッチポートは、localnet VLAN構成を正しく機能させるために**トランクモードで構成されている必要があります**:

* **トランクモード**: スイッチポートが複数のVLANのトラフィックを運ぶことを許可します
* **VLANタグ付け**: 各VLANは一意のVLAN ID（例: 100, 200, 300）によって識別されます
* **複数のネットワーク**: 複数のVLANベースのセカンダリネットワークをサポートするために不可欠です

== 手順 1: VMの名前空間の作成

VLAN構成付きのlocalnetセカンダリネットワークを使用するVM用の名前空間を作成します。

xref:attachment$localnet-vlan/localnet-vlan-namespace.yaml[localnet-vlan-namespace.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: vm-guests-vlan
EOF
----

== 手順 2: ブリッジマッピングの構成

論理ネットワークインターフェース名 `localnet-vlan` をワーカーノード上の `br-ex` ブリッジインターフェースに接続するブリッジマッピングを構成します。このマッピングにより、OVN-Kubernetesに対してどの物理ブリッジインターフェースがVLANサポート付きのlocalnetトラフィックを処理するかを通知し、基盤となる物理ネットワークインフラストラクチャ経由でVMがVLANタグ付き外部ネットワークにアクセスできるようにします。

xref:attachment$localnet-vlan/localnet-vlan-bridge-mapping-nncp.yaml[localnet-vlan-bridge-mapping-nncp.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: localnet-vlan-bridge-mapping
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ''  
  desiredState:
    ovn:
      bridge-mappings:
      - localnet: localnet-vlan
        bridge: br-ex
        state: present
EOF
----

== 手順 3: VLAN付きNetworkAttachmentDefinitionの作成

セカンダリネットワーク接続用に、VLANタグ付けを伴うlocalnetトポロジーを構成するNetworkAttachmentDefinitionを作成します。

xref:attachment$localnet-vlan/localnet-vlan-100-nad.yaml[localnet-vlan-100-nad.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: "k8s.cni.cncf.io/v1"
kind: NetworkAttachmentDefinition
metadata:
  name: localnet-vlan-100
  namespace: vm-guests-vlan
spec:
  config: '{
    "cniVersion": "0.3.1",
    "name": "localnet-vlan-100",
    "type": "ovn-k8s-cni-overlay",
    "topology": "localnet",
    "netAttachDefName": "vm-guests-vlan/localnet-vlan-100",
    "physicalNetworkName": "localnet-vlan",
    "vlanID": 100,
    "subnets": "192.168.100.0/24",
    "excludeSubnets": "192.168.100.1/32"
  }'
EOF
----

**構成の詳細**:

* `physicalNetworkName: "localnet-vlan"`: このNetworkAttachmentDefinition (NAD) をNNCPで構成されたブリッジマッピングにリンクします（bridge-mappings内の `localnet` 名と一致させる必要があります）
* `vlanID: 100`: ネットワークセグメンテーション用にVLANタグ100を指定します
* `subnets`: OVN-Kubernetes IPAM用のIPサブネット範囲を定義します
* `excludeSubnets`: ゲートウェイIPが割り当てられないように除外します（通常はルーター/ゲートウェイ用に予約されます）
* `topology: "localnet"`: 物理ネットワークへの直接アクセスのためにlocalnetトポロジーを使用します

== 手順 4: VLANネットワーク接続を備えたVMのデプロイ

PodネットワークとVLANタグ付きlocalnetセカンダリネットワークの両方に接続するVMを作成します。

WARNING: この例のcloud-initパスワードはデモ目的のみのものです。本番環境では、常に強力で一意なパスワード、またはSSHキー認証を使用してください。

xref:attachment$localnet-vlan/fedora-vlan-vm.yaml[fedora-vlan-vm.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vlan-vm
  namespace: vm-guests-vlan
  labels:
    app: fedora-vlan-vm
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-vlan-volume
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 33Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        app: fedora-vlan-vm
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            bridge: {}
          - name: vlan_network
            bridge: {}
        resources:
          requests:
            memory: 2Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      - name: vlan_network
        multus:
          networkName: localnet-vlan-100
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: fedora-vlan-volume
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            user: fedora
            password: fedora
            chpasswd: { expire: False }
            packages:
            - python3
            - tcpdump
            - net-tools
            runcmd:
            - echo "<h1>Welcome to OpenShift Virtualization VLAN Network!</h1>" > /root/index.html
            - echo "<p>VLAN ID: 100</p>" >> /root/index.html
            - echo "<p>Network: 192.168.100.0/24</p>" >> /root/index.html
            - cd /root && nohup python3 -m http.server 8080 > /dev/null 2>&1 &
EOF
----

== 手順 5: VLANネットワーク接続の確認

VMコンソールにアクセスし、VLAN構成によるデュアルネットワーク接続を確認します。

[source,bash,role=execute]
----
# Check the VM status
oc get vm -n vm-guests-vlan

# Check the running VM instance
oc get vmi -n vm-guests-vlan

# Access the VM console (credentials: fedora/fedora via cloud-init)
virtctl console -n vm-guests-vlan fedora-vlan-vm
----

VMコンソール内で、VLANネットワーク構成を確認します:

[source,bash,role=execute]
----
# Check network interfaces - primary (pod network) and secondary (VLAN network)
ip addr show

# Check routes - two default gateways (lower metric prioritized)
# Primary network: metric 100, Secondary VLAN: metric 101
ip route show

# Expected output example:
default via 10.128.0.1 dev enp1s0 proto dhcp src 10.128.0.172 metric 100 
default via 192.168.100.1 dev enp2s0 proto dhcp src 192.168.100.50 metric 101 

# Test external connectivity
ping 8.8.8.8

# Verify the web service is running
curl localhost:8080

# Test VLAN interface connectivity (replace <vlan-ip> with actual IP)
curl <vlan-ip>:8080

# Verify VLAN tagging (if tcpdump is available)
# This will show VLAN-tagged traffic on the secondary interface
sudo tcpdump -i enp2s0 -nn vlan 100
----

== トラブルシューティング

=== 一般的な問題

. **VMまでVLANトラフィックが到達しない**:
   * 物理スイッチ構成がVLANタグ付けをサポートしているか確認する
   * ワーカーノードのブリッジマッピング構成を確認する
   * VLAN IDがネットワークインフラストラクチャと一致していることを確認する

. **IPアドレスの割り当て問題**:
   * VLANネットワーク上でDHCPサーバーが利用可能であることを確認する

. **ネットワーク接続の問題**:
   * 物理ネットワークが構成されたVLANをサポートしていることを確認する
   * 物理ネットワークからVLANサブネットへの接続をテストする

=== 確認コマンド

[source,bash,role=execute]
----
# Check NetworkAttachmentDefinition status
oc get network-attachment-definitions -n vm-guests-vlan

# Verify bridge mapping on worker nodes
oc get nncp localnet-vlan-bridge-mapping -o yaml

# Check VM network status
oc get vmi -n vm-guests-vlan -o yaml | grep -A 10 networks

# View OVN-Kubernetes logs for troubleshooting
oc logs -n openshift-ovn-kubernetes -l app=ovnkube-node
----

== まとめ

本チュートリアルでは、以下の内容を学習しました:

* ワーカーノード上で `NodeNetworkConfigurationPolicy` を使用して、VLANサポートを備えたブリッジマッピングを構成する方法
* localnetトポロジー、VLANタグ付け、およびOVN-Kubernetes IPAMを備えた `NetworkAttachmentDefinition` を作成する方法
* VLANタグ付きトラフィックをサポートするために物理スイッチポートをトランクモードで構成する必要がある理由
* PodネットワークとVLANタグ付きlocalnetを組み合わせたデュアルネットワーク接続を備えたVMをデプロイする方法
* VLANネットワーク構成を検証し、一般的なVLAN接続の問題をトラブルシューティングする方法
