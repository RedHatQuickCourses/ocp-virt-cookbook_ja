= HostPath Provisioner を使用したローカルストレージ
:navtitle: HostPath Provisioner

== 概要

HostPath Provisioner (HPP) は、OpenShift Virtualization ワークロードに対してローカルノードストレージを使用するシンプルな方法を提供します。HPP は OpenShift Virtualization オペレーターの一部として含まれており、ホストファイルシステムのディレクトリをバックエンドとする永続ボリューム (Persistent Volume) を作成します。

=== HostPath Provisioner の使用場面

HPP は以下のような場合に適しています。

* **開発およびテスト環境**: 複雑なストレージインフラを構築せずに素早くセットアップ可能
* **シングルノードクラスター**: 分散ストレージが不要な環境
* **プロトタイピング**: 本番環境向けのストレージを導入する前の迅速な反復作業
* **エッジデプロイメント**: インフラ要件が最小限である環境

== 前提条件

* **OpenShift 4.21+** (OpenShift Virtualization がインストール済みであること)
* **クラスター管理者アクセス**: HostPathProvisioner および StorageClass リソースを作成するために必要
* **利用可能なローカルディスク容量**: VM ストレージとして十分な容量がワーカーノードにあること
* **oc CLI**: インストールおよび設定済みであること
* **virtctl CLI**: VM コンソールアクセスに必要

OpenShift Virtualization がインストールされていることを確認します。

[source,bash,role=execute]
----
oc get csv -n openshift-cnv | grep kubevirt
----

オペレーターのバージョンが表示されれば成功です（バージョンは環境により異なります）。
----
kubevirt-hyperconverged-operator.v4.x.x   OpenShift Virtualization   4.x.x   Succeeded
----

== ステップ 1: ホストディレクトリの準備

HPP は、ボリュームを格納する各ノード上のディレクトリを必要とします。このディレクトリが存在し、適切な SELinux コンテキストを持っている必要があります。

=== オプション A: MachineConfig の使用（本番環境で推奨）

MachineConfig を作成し、すべてのワーカーノードでディレクトリが存在し、再起動後も永続化されるようにします。

xref:attachment$hostpath-provisioner/hpp-machineconfig.yaml[hpp-machineconfig.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 50-hpp-directory
  labels:
    machineconfiguration.openshift.io/role: worker
spec:
  config:
    ignition:
      version: 3.2.0
    systemd:
      units:
      - name: create-hpp-directory.service
        enabled: true
        contents: |
          [Unit]
          Description=Create HostPath Provisioner directory
          Before=kubelet.service

          [Service]
          Type=oneshot
          ExecStart=/bin/mkdir -p /var/hpvolumes
          ExecStart=/bin/chcon -Rt container_file_t /var/hpvolumes
          RemainAfterExit=yes

          [Install]
          WantedBy=multi-user.target
EOF
----

注意: MachineConfig を適用すると、ワーカーノードのローリング再起動がトリガーされます。すべてのノードが更新されるまで待機してから進んでください。

MachineConfig のロールアウトを監視します。

[source,bash,role=execute]
----
oc get mcp worker
----

`UPDATED` が `True`、`UPDATING` が `False` になるまで待ちます。
----
NAME     CONFIG                                             UPDATED   UPDATING   DEGRADED   MACHINECOUNT   READYMACHINECOUNT
worker   rendered-worker-abc123                             True      False      False      6              6
----

=== オプション B: 手動でのディレクトリ作成（クイックテスト用）

クイックテストを行う場合は、各ノードで手動でディレクトリを作成します。

まず、ワーカーノードのリストを取得します。

[source,bash,role=execute]
----
oc get nodes -l node-role.kubernetes.io/worker
----

次に、各ワーカーノードに対して以下のコマンドを実行します（`<worker-node-name>` は実際のノード名に置き換えてください）。

[source,bash,role=execute]
----
oc debug node/<worker-node-name> -- chroot /host mkdir -p /var/hpvolumes
----

[source,bash,role=execute]
----
oc debug node/<worker-node-name> -- chroot /host chcon -Rt container_file_t /var/hpvolumes
----

注意: 手動でのディレクトリ作成は、ノードの再起動後には永続化されません。永続的な設定には MachineConfig を使用してください。

=== ディレクトリ作成の確認

ディレクトリが存在し、適切な SELinux コンテキストが設定されていることを確認します。

[source,bash,role=execute]
----
oc debug node/<worker-node-name> -- chroot /host ls -laZ /var/hpvolumes
----

出力例:
----
total 0
drwxr-xr-x. 2 root root system_u:object_r:container_file_t:s0 6 Mar 19 10:00 .
drwxr-xr-x. 22 root root system_u:object_r:var_t:s0          4096 Mar 19 10:00 ..
----

`container_file_t` という SELinux コンテキストにより、コンテナはこのディレクトリに対して読み書きが可能になります。

== ステップ 2: HostPathProvisioner の作成

HostPathProvisioner カスタムリソースを作成して、CSI ドライバーを設定します。

xref:attachment$hostpath-provisioner/hpp-hostpathprovisioner.yaml[hpp-hostpathprovisioner.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: hostpathprovisioner.kubevirt.io/v1beta1
kind: HostPathProvisioner
metadata:
  name: hostpath-provisioner
spec:
  imagePullPolicy: IfNotPresent
  storagePools:
  - name: local
    path: /var/hpvolumes
  workload:
    nodeSelector:
      kubernetes.io/os: linux
EOF
----

=== 設定パラメーター

* **`storagePools`**: 1 つ以上のストレージプールを定義します（それぞれに固有の名前とパスを指定）。
** `name`: StorageClass でこのプールを参照するために使用する識別子
** `path`: ボリュームが作成されるホストノード上のディレクトリ
* **`workload.nodeSelector`**: HPP CSI ポッドを実行するノードを制御します。
* **`imagePullPolicy`**: コンテナイメージのプルポリシー。

=== HostPathProvisioner の確認

HostPathProvisioner のステータスを確認します。

[source,bash,role=execute]
----
oc get hostpathprovisioner hostpath-provisioner
----

出力例:
----
NAME                   AGE
hostpath-provisioner   30s
----

CSI ポッドがワーカーノードで実行されていることを確認します。

[source,bash,role=execute]
----
oc get pods -n openshift-cnv -l k8s-app=hostpath-provisioner
----

出力例（ワーカーノードごとに 1 つのポッド）:
----
NAME                                           READY   STATUS    RESTARTS   AGE
hostpath-provisioner-csi-xxxxx                 4/4     Running   0          2m
hostpath-provisioner-csi-yyyyy                 4/4     Running   0          2m
hostpath-provisioner-csi-zzzzz                 4/4     Running   0          2m
----

== ステップ 3: StorageClass の作成

HostPath Provisioner を使用する StorageClass を作成します。

xref:attachment$hostpath-provisioner/hpp-storageclass.yaml[hpp-storageclass.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: hostpath-csi
provisioner: kubevirt.io.hostpath-provisioner
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
parameters:
  storagePool: local
EOF
----

=== 設定パラメーター

* **`provisioner`**: `kubevirt.io.hostpath-provisioner` である必要があります。
* **`reclaimPolicy`**: `Delete` に設定すると、PVC が削除されたときに PV データも削除されます。データを保持するには `Retain` を使用します。
* **`volumeBindingMode`**: `WaitForFirstConsumer` は、ポッドが PVC を使用するまで PV の作成を遅延させ、正しいノード上にボリュームが作成されるようにします。
* **`parameters.storagePool`**: HostPathProvisioner で定義されたストレージプール名を参照します。

=== StorageClass の確認

[source,bash,role=execute]
----
oc get storageclass hostpath-csi
----

出力例:
----
NAME           PROVISIONER                        RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
hostpath-csi   kubevirt.io.hostpath-provisioner   Delete          WaitForFirstConsumer   false                  10s
----

=== オプション: デフォルト StorageClass への設定

HPP をデフォルトの StorageClass に設定します（他のストレージがない開発/テストクラスターで有用です）。

[source,bash,role=execute]
----
oc patch storageclass hostpath-csi -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
----

注意: クラスター内にはデフォルトの StorageClass は 1 つだけであるべきです。他にデフォルトが存在する場合は、先にそのアノテーションを削除してください。

[source,bash,role=execute]
----
# 既存の StorageClass からデフォルト設定を削除
oc patch storageclass <existing-default> -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"false"}}}'
----

== ステップ 4: ストレージプロビジョニングのテスト

テスト用の PVC を作成し、HPP が機能していることを確認します。

xref:attachment$hostpath-provisioner/hpp-test-pvc.yaml[hpp-test-pvc.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: hpp-test-pvc
  namespace: default
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: hostpath-csi
EOF
----

PVC のステータスを確認します。

[source,bash,role=execute]
----
oc get pvc hpp-test-pvc
----

`WaitForFirstConsumer` の場合、PVC はポッドが使用するまで `Pending` 状態のままです。
----
NAME           STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   VOLUMEATTRIBUTESCLASS   AGE
hpp-test-pvc   Pending                                      hostpath-csi   <unset>                 5s
----

ボリュームのバインディングをトリガーするためにテストポッドを作成します。

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: hpp-test-pod
  namespace: default
spec:
  containers:
  - name: test
    image: registry.access.redhat.com/ubi9/ubi:latest
    command: ["sleep", "3600"]
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: hpp-test-pvc
EOF
----

PVC がバインドされたことを確認します。

[source,bash,role=execute]
----
oc get pvc hpp-test-pvc
----

出力例:
----
NAME           STATUS   VOLUME                                     CAPACITY   ACCESS MODES   STORAGECLASS   VOLUMEATTRIBUTESCLASS   AGE
hpp-test-pvc   Bound    pvc-abc12345-6789-def0-1234-567890abcdef   5Gi        RWO            hostpath-csi   <unset>                 30s
----

注意: HPP は、要求されたサイズではなく、ホストパス上の全利用可能容量を報告する場合があります。これは hostpath ベースのストレージにおける期待される動作です。

ボリュームへの書き込みテストを行います。

[source,bash,role=execute]
----
oc exec hpp-test-pod -- sh -c "echo 'HPP Storage Test' > /data/test.txt && cat /data/test.txt"
----

出力例:
----
HPP Storage Test
----

テストリソースをクリーンアップします。

[source,bash,role=execute]
----
oc delete pod hpp-test-pod
oc delete pvc hpp-test-pvc
----

== ステップ 5: 仮想マシンでの使用

HPP ベースのストレージを使用する VM を作成します。

警告: この例の cloud-init パスワードはデモ目的のみです。本番環境では、常に強力で一意のパスワードを使用するか、SSH キー認証を使用してください。

xref:attachment$hostpath-provisioner/hpp-test-vm.yaml[hpp-test-vm.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: hpp-test-vm
  namespace: default
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        kubevirt.io/vm: hpp-test-vm
    spec:
      domain:
        cpu:
          cores: 1
        devices:
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            masquerade: {}
        resources:
          requests:
            memory: 1Gi
      networks:
      - name: default
        pod: {}
      volumes:
      - name: rootdisk
        dataVolume:
          name: hpp-test-vm-rootdisk
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            user: fedora
            password: fedora
            chpasswd:
              expire: false
  dataVolumeTemplates:
  - metadata:
      name: hpp-test-vm-rootdisk
    spec:
      storage:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 10Gi
        storageClassName: hostpath-csi
      source:
        registry:
          url: docker://quay.io/containerdisks/fedora:latest
EOF
----

VM と DataVolume のステータスを監視します。

[source,bash,role=execute]
----
# DataVolume のインポート進捗を確認
oc get dv -n default hpp-test-vm-rootdisk -w
----

進行状況の例:
----
NAME                    PHASE              PROGRESS   RESTARTS   AGE
hpp-test-vm-rootdisk    ImportScheduled    N/A                   5s
hpp-test-vm-rootdisk    ImportInProgress   0.00%                 10s
hpp-test-vm-rootdisk    ImportInProgress   45.50%                30s
hpp-test-vm-rootdisk    Succeeded          100.0%                1m
----

VM のステータスを確認します。

[source,bash,role=execute]
----
oc get vm hpp-test-vm
oc get vmi hpp-test-vm
----

出力例:
----
NAME          AGE   STATUS    READY
hpp-test-vm   2m    Running   True

NAME          AGE   PHASE     IP             NODENAME   READY
hpp-test-vm   2m    Running   10.x.x.x       workerXX   True
----

VM コンソールにアクセスします。

[source,bash,role=execute]
----
virtctl console hpp-test-vm
----

VM リソースをクリーンアップします。

[source,bash,role=execute]
----
oc delete vm hpp-test-vm
----

== トラブルシューティング

=== PVC が Pending のままになる

ポッド作成後も PVC が Pending の場合:

[source,bash,role=execute]
----
# PVC イベントを確認
oc describe pvc <pvc-name>

# HPP ポッドのログを確認
oc logs -n openshift-cnv -l k8s-app=hostpath-provisioner -c hostpath-provisioner
----

一般的な原因:

* **ディレクトリが存在しない**: 対象ノード上で `/var/hpvolumes` が存在することを確認してください。
* **SELinux コンテキストが不適切**: `chcon -Rt container_file_t /var/hpvolumes` が適用されたか確認してください。
* **HPP ポッドが実行されていない**: HPP CSI ポッドがすべてのワーカーノードで実行されていることを確認してください。

=== 権限拒否 (Permission Denied) エラー

ボリュームへの書き込み時に権限エラーが発生する場合:

[source,bash,role=execute]
----
# ホスト上の SELinux コンテキストを確認
oc debug node/<worker-node-name> -- chroot /host ls -laZ /var/hpvolumes

# 必要に応じて SELinux コンテキストを再適用
oc debug node/<worker-node-name> -- chroot /host chcon -Rt container_file_t /var/hpvolumes
----

=== HPP ポッドが起動しない

HostPathProvisioner のステータスとイベントを確認してください。

[source,bash,role=execute]
----
oc describe hostpathprovisioner hostpath-provisioner

# openshift-cnv 名前空間の問題を確認
oc get events -n openshift-cnv --sort-by='.lastTimestamp' | tail -20
----

=== ボリュームが期待したノードに作成されない

`WaitForFirstConsumer` の場合、ボリュームはポッドがスケジュールされたノード上に作成されます。ノードの配置を制御したい場合:

[source,yaml]
----
# ポッドスペックに nodeSelector を追加
spec:
  nodeSelector:
    kubernetes.io/hostname: <worker-node-name>
----

== クリーンアップ

クラスターから HPP を削除するには:

[source,bash,role=execute]
----
# StorageClass の削除
oc delete storageclass hostpath-csi

# HostPathProvisioner の削除
oc delete hostpathprovisioner hostpath-provisioner

# オプション: MachineConfig の削除 (ノードの再起動がトリガーされます)
oc delete machineconfig 50-hpp-directory
----

注意: HostPathProvisioner を削除しても、既存の PV は削除されません。すべてのデータをクリーンアップしたい場合は、先に PVC を削除してください。
